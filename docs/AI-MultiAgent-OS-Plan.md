# 桌面端 AI 多智能体操作系统重构计划

> 适用项目：`zyfront-desktop`
>
> 重构目标：将当前“调试面板式”交互，升级为可用于开发、协作、部署的 AI 多智能体操作系统。
>
> 本文只讨论功能、流程、角色、模块职责、信息展示与操作逻辑，不涉及视觉美化。

---

## 1. 目标定义

### 1.1 产品定位

本产品由两层组成：

- **终端页**：命令执行层（CLI 会话层）
- **多 Agent 看板**：智能体管理层（大脑层）

最终希望形成一个统一的“团队编排 + 任务执行 + 会话观测 + 异常恢复”的桌面工作台。

### 1.2 核心原则

1. 终端页只做命令执行，不做智能体决策。
2. 多 Agent 看板只做角色管理、任务编排与团队控制。
3. 用户看到的是“流畅模式 / 保活模式 / 静默模式”，而不是内部 backend 术语。
4. 所有状态必须可理解、可操作、可恢复。
5. 同一职责只允许在一个页面出现一次，避免功能重复。

---

## 2. 现状问题总结

当前问题可归纳为五类：

### 2.1 页面职责混乱

终端页混入了 Agent 控制、任务分配、观测信息等非终端职责；看板页又混入了底层执行细节，导致用户不知道应该在什么地方完成什么动作。

### 2.2 状态不可理解

内部状态如 backend detection、effectiveBackend、configuredMode 对用户没有直接意义，且容易造成认知负担。

### 2.3 操作路径过长

用户从创建 Team、创建 Agent、绑定会话、分配任务、查看输出到恢复异常，路径断裂，缺少标准主线。

### 2.4 角色职责不清晰

Leader、Teammate、Team 三层职责存在交叉，导致“谁负责决策、谁负责执行、谁负责全局管控”不明确。

### 2.5 恢复机制不统一

事件流、重试、快照、tmux 修复散落在不同位置，用户面对错误时无法快速判断下一步操作。

## 3. 信息架构重构

### 3.1 终端页 = 命令执行层

终端页仅保留以下三类功能：

1. **多 Tab 会话管理**
   - 新建
   - 切换
   - 关闭
   - Attach
   - Detach

2. **实时 Shell 输出**
   - 命令输出
   - 错误输出
   - 交互式输入回显

3. **会话控制**
   - 重连
   - 重启 tmux 会话
   - 终止当前会话

终端页移除项：
- Agent 角色选择
- 任务分配
- 团队观测面板
- 模式切换细节
- 复杂配置项

### 3.2 多 Agent 看板 = 智能体管理层

看板拆成 4 个功能区：

#### A. 团队控制区
- 运行模式选择
- 环境可用性显示
- 全局启动 / 停止 / 快照 / 恢复

#### B. 智能体列表区
- 角色
- 状态
- 后端
- 会话绑定
- 任务信息
- 单 Agent 控制按钮

#### C. 任务流协作区
- 创建任务
- 指定执行者
- Leader 下发
- Teammate 执行
- 任务状态流转

#### D. 观测与恢复区
- 事件流
- 重试记录
- 快照管理
- tmux 异常修复

## 4. 页面功能边界

### 4.1 终端页边界

终端页负责：
- Shell 命令执行
- 实时输出
- 会话切换
- Attach / Detach
- 当前会话存活状态

终端页不负责：
- 任务规划
- Agent 决策
- 团队编排
- 快照管理
- 事件聚合

### 4.2 多 Agent 看板边界

看板页负责：
- Agent 生命周期管理
- 角色管理
- 任务分配
- 会话映射
- 协作状态展示
- 异常恢复入口

看板页不负责：
- 用户逐条输入命令
- 实时 shell 输出滚动展示
- 具体命令执行交互

### 4.3 边界总结

- 终端页回答“**我在执行什么**”
- 看板页回答“**谁在执行、为什么执行、执行得怎么样**”

## 5. 角色体系定义

### 5.1 Leader Agent

#### 定位
Leader 是团队大脑，只负责规划、拆分、分配、汇总，不直接承载终端会话。

#### 后端约束
- 固定为 `in-process`
- 不占终端 Tab
- 不展示 shell 交互入口

#### 职责
- 任务规划
- 任务拆分
- 分配 Teammate
- 汇总执行结果
- 决策重试与恢复

### 5.2 Teammate Agent

#### 定位
Teammate 是执行者，负责运行任务、调用工具、执行命令。

#### 后端约束
- `tmux`：绑定终端 Tab，可 Detach / Attach
- `embedded-gui`：内置交互式执行界面
- `in-process`：静默执行，不进入终端页

#### 规则
- 1 个 Teammate = 1 个执行会话
- 必须绑定明确后端
- 必须可追踪到会话 ID

### 5.3 Team

#### 定位
Team 是全局编排单元，负责资源、会话、快照、恢复和协作总控。

#### 职责
- 管理 Leader + Teammates
- 管理运行模式
- 管理会话生命周期
- 管理事件与快照
- 管理恢复动作

## 6. 模式体系重命名

### 6.1 对用户展示的模式名称

| 用户可见名称 | 内部映射 | 含义 |
|---|---|---|
| 流畅模式 | embedded-gui | 交互顺滑、内置会话、无需系统终端窗口 |
| 保活模式 | tmux（WSL2） | 后台持续运行，可断线重连，适合长任务 |
| 静默模式 | in-process | 后台执行，不打开终端，不打扰用户 |

### 6.2 环境约束

- tmux 仅在 WSL2 可用时启用
- WSL2 不可用时自动 fallback 到 embedded-gui
- tmux 不可用时必须显式灰掉或不可选

## 9. 标准用户操作路径

### 9.1 从 0 开始跑一个任务

1. 创建 Team，系统自动生成 Leader。
2. 系统检测环境并建议运行模式。
3. 添加 Teammate，自动创建对应执行会话。
4. 创建任务并指定 Teammate。
5. Leader 规划并下发，Teammate 执行。
6. 用户可随时 Attach 到对应终端查看输出。
7. 任务完成后归档结果并保存快照。

### 9.2 异常恢复路径

#### tmux 断开
- 状态显示为“后台保活”
- 用户点击“连接”或“重连”即可恢复

#### tmux 异常
- 状态显示为“异常”
- 用户点击“修复 tmux”
- 系统重建会话并重新绑定 Agent

#### 任务失败
- 显示可理解的失败原因
- 提供“重试 / 重新分配 / 查看事件”入口

## 10. 里程碑计划

- 去除内部实现术语的直接暴露

#### 任务列表

- [ ] 梳理多 Agent 看板现有功能，标记保留/移除/迁移项
- [ ] 统一页面文案中的模式名称、状态名称、按钮名称

- [ ] 删除或隐藏看板页中的底层命令输出与终端级控制项

## 12. 关键设计约束

1. Leader 不进入终端页。
2. Teammate 必须和会话绑定。
3. 终端页不承载任务编排。
4. 看板页不承载 shell 交互。
5. 用户可见的状态必须是业务语言。
6. 所有异常必须有恢复动作。
7. 所有 Agent 行为必须能追踪到事件。
8. 快照必须覆盖 Team 全局状态。

---

## 13. 落地建议

### 第一阶段
先做页面职责收敛和状态重命名，确保信息结构先变干净。

### 第二阶段
重建 Agent 与会话映射，确保 Leader / Teammate 分工稳定。

### 第三阶段
重建任务流与观测恢复体系，形成可实际使用的 AI 协作工作台。

### 第四阶段
做默认路径优化，压缩用户从进入页面到跑通第一任务的步骤数。

---

## 14. 结论

本次重构的核心不是增加更多功能，而是将现有能力重新分层：

- 终端负责执行
- 看板负责调度
- Leader 负责决策
- Teammate 负责执行
- Team 负责统筹

只要这五层关系清楚，整个桌面端就能从“调试界面”升级为“可协作、可恢复、可持续使用的 AI 多智能体操作系统”。
