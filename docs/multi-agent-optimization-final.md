# 当前协作页接入多智能体系统的最终优化文档

> 适用项目：`zyfront-desktop`
>
> 目标：在现有协作页与看板页基础上，接入可落地的多智能体操作系统能力，形成“可编排、可对抗、可观测、可恢复、可视化”的统一协作中枢。
>
> 本文聚焦产品定义、交互边界、数据模型、状态流转、对抗模式、手动编排、自动编排、可视化方案与实施路径。

---

## 1. 背景与目标

当前 `AI-MultiAgent-OS-Plan.md` 已明确终端页与看板页的职责边界，但协作页仍需要进一步升级：

- 从“任务展示页”升级为“多智能体编排控制台”
- 从“单一协作流程”升级为“多模式协作系统”
- 从“人工驱动执行”升级为“手动编排 + 自动编排并存”
- 从“状态列表”升级为“实时可视化协作网络”

### 1.1 最终目标

构建一个统一的多智能体系统能力层，支持：

1. **5 种对抗模式**，满足不同协作/竞争场景
2. **手动编排多智能体**，让用户可精确控制团队结构与任务流
3. **自动编排多智能体**，让系统根据目标自动完成组队、分工、调度与收敛
4. **实时可视化**，把智能体关系、任务状态、对抗过程和结果输出清楚地展示出来
5. **异常恢复与追踪**，确保每个步骤都可理解、可回放、可修复

---

## 2. 总体设计原则

### 2.1 统一抽象

所有协作场景都统一抽象为以下四层：

- **Team**：全局编排单元，负责资源、规则、模式与生命周期
- **Agent**：具体执行或决策单元
- **Task**：协作目标拆分后的执行单元
- **Session**：Agent 对应的运行会话或执行后端

### 2.2 统一显示语言

对用户只展示业务语言，不暴露内部实现术语：

- 使用“流畅模式 / 保活模式 / 静默模式”替代后端术语
- 使用“启动 / 连接 / 重试 / 重新分配 / 恢复”替代低层命令概念
- 使用“对抗 / 协作 / 编排 / 收敛”替代纯技术状态名

### 2.3 统一操作路径

协作页最终只保留三种核心动作：

1. **发起任务**：用户输入目标或任务
2. **选择编排方式**：手动或自动
3. **查看结果与回放**：实时观测、对抗过程、结果归档

---

## 3. 协作页在系统中的定位

协作页不是终端页，也不是纯看板页，而是两者之间的“协作控制层”。

### 3.1 与终端页的关系

- 终端页只负责执行
- 协作页负责决定谁执行、如何执行、是否重试、如何收敛
- 协作页可跳转到具体 Agent 对应的终端会话

### 3.2 与看板页的关系

- 看板页侧重团队与 Agent 生命周期管理
- 协作页侧重任务流、对抗机制与结果产出
- 看板是“团队状态面板”，协作页是“协作作战中枢”

### 3.3 与系统编排服务的关系

协作页直接对接 `MultiAgentOrchestratorService` / `TeamRuntimeService` / `SessionMappingService` 一类服务，完成：

- 团队创建
- Agent 生成
- 会话绑定
- 任务拆解
- 运行模式选择
- 事件订阅与回放

## 4. 5 种对抗模式设计

> 这里的“对抗模式”并不只表示打架，而是指五种不同的多智能体竞争/比较/收敛机制。

### 4.1 模式总览

| 模式 | 核心目标 | 适用场景 | 输出结果 |
|---|---|---|---|
| 辩论对抗模式 | 通过观点碰撞得出最优答案 | 方案评审、架构争议 | 最终建议 + 胜负评分 |
| 红蓝攻防模式 | 从攻击/防御两端验证方案鲁棒性 | 安全评估、边界测试 | 漏洞点 + 防御建议 |
| 竞赛冲刺模式 | 多 Agent 争抢任务目标，比较效率 | 需求交付、性能优化 | 排名、耗时、质量分 |
| 回合博弈模式 | 按轮次决策，逐步逼近最优解 | 策略推演、复杂决策 | 回合轨迹 + 赢家 |
| 混合评审模式 | 综合辩论、投票、评分、裁判收敛 | 方案决策、产品评审 | 最终裁定 + 共识结果 |

---

### 4.2 模式一：辩论对抗模式

#### 目标
让多个 Agent 围绕同一个问题进行正反观点输出，通过评分与裁决选出更优方案。

#### 流程
1. 用户输入议题
2. 系统创建正方/反方/裁判 Agent
3. 各 Agent 输出论点
4. 裁判 Agent 评估论点质量
5. 输出最终结论

#### 关键规则
- 每轮只允许一个 Agent 发言
- 发言内容必须绑定证据来源或推理链
- 裁判可打断、追问或要求补充

#### 可视化要点
- 中心辩题卡
- 左右阵营面板
- 轮次进度条
- 论点碰撞连线动画
- 裁决结果高亮

---

### 4.3 模式二：红蓝攻防模式

#### 目标
模拟攻击方与防御方围绕某个方案、配置或策略展开攻防，验证系统鲁棒性。

#### 流程
1. 用户选择攻防主题
2. 红队生成攻击策略
3. 蓝队生成防御方案
4. 裁判记录漏洞与防守缺口
5. 输出风险清单与加固建议

#### 关键规则
- 红队优先暴露问题
- 蓝队优先修复问题
- 裁判记录“漏洞 -> 证据 -> 防御 -> 结果”链路

#### 可视化要点
- 红蓝双阵营卡片
- 风险等级条
- 攻击路径图
- 防御覆盖图
- 风险热力图

---

### 4.4 模式三：竞赛冲刺模式

#### 目标
多个 Agent 同时围绕同一目标冲刺，比较速度、质量、成本、稳定性。

#### 流程
1. 用户定义目标与评价维度
2. 系统为多个 Agent 分配同一目标
3. 各 Agent 独立产出结果
4. 系统根据评分模型自动排序
5. 选出最优方案

#### 关键规则
- 所有 Agent 接收同一初始约束
- 输出必须可量化比较
- 评分维度至少包含：速度、质量、完整性、成本

#### 可视化要点
- 跑道式排名条
- 实时分数榜
- 阶段冲刺进度
- 时间消耗对比
- 冠军结果卡

---

### 4.5 模式四：回合博弈模式

#### 目标
用回合制机制让多个 Agent 逐步调整策略，观察策略演化过程。

#### 流程
1. 用户输入目标与初始约束
2. 系统按轮次分配发言顺序
3. 每轮 Agent 基于上一轮结果更新策略
4. 收敛后由裁判给出最终输出

#### 关键规则
- 每轮必须继承上一轮上下文
- 允许“反转策略”“补充策略”“否定策略”
- 终止条件可由用户手动控制或系统自动判断

#### 可视化要点
- 回合时间轴
- 当前轮次高亮
- 决策树展开图
- 历史策略卡片
- 收敛状态指示器

---

### 4.6 模式五：混合评审模式

#### 目标
融合辩论、竞赛、投票与裁决，形成适合最终决策的综合模式。

#### 流程
1. 多 Agent 提交不同方案
2. 系统组织讨论
3. 用户或裁判 Agent 投票
4. 综合评分
5. 输出最终一致意见或保留分歧记录

#### 关键规则
- 支持人工投票 + AI 裁决双轨制
- 对分歧项必须保留原因
- 最终输出需给出“推荐方案 + 备选方案 + 不采用原因”

#### 可视化要点
- 方案卡片墙
- 投票条形图
- 分歧点标签
- 最终推荐摘要

---

## 5. 手动编排多智能体模式

手动编排适合专家用户、复杂项目或需要强控制的场景。

### 5.1 设计目标

- 用户可以明确指定每个 Agent 的角色、能力、后端、权重与任务
- 用户可以人工控制任务流与执行顺序
- 用户可以在任意阶段插入、暂停、替换、重试 Agent

### 5.2 手动编排支持的能力

#### 1）团队创建
- 新建 Team
- 指定 Team 名称、目标、运行模式
- 选择对抗模式

#### 2）Agent 配置
- 指定角色：Leader / Teammate / Judge / Red / Blue / Reviewer
- 指定后端：流畅模式 / 保活模式 / 静默模式
- 指定能力标签：规划、编码、分析、审查、测试、总结

#### 3）任务编排
- 手工创建任务节点
- 手动设置依赖关系
- 手动指定执行者
- 手动设置优先级和截止条件

#### 4）执行控制
- 启动
- 暂停
- 跳过
- 重新分配
- 强制结束
- 重连会话

### 5.3 手动编排交互流程

1. 用户在协作页选择“手动编排”
2. 系统展示 Team 画布
3. 用户拖拽 Agent 到任务槽位
4. 用户设置顺序和依赖
5. 用户点击“启动编排”
6. 系统按手动规则执行并实时回传
7. 用户可随时调整并继续

### 5.4 手动编排的数据结构建议

```ts
interface ManualPlan {
  teamId: string;
  mode: 'battle' | 'red-blue' | 'sprint' | 'turn-based' | 'review';
  nodes: ManualNode[];
  edges: ManualEdge[];
  policy: {
    allowPause: boolean;
    allowReassign: boolean;
    allowHumanOverride: boolean;
  };
}
```

### 5.5 手动编排适用场景

- 大型重构任务
- 架构方案评审
- 安全验证
- 需要人工强监管的流程

---

## 6. 自动编排多智能体模式

自动编排适合大多数常规任务，系统负责自动拆解、分工、调度与收敛。

### 6.1 设计目标

- 输入一个目标，系统自动生成团队结构
- 系统自动选择适合的模式
- 系统自动分配角色与执行顺序
- 系统自动监控、调整和收敛

### 6.2 自动编排核心能力

#### 1）目标解析
把用户输入转换成：
- 目标类型
- 复杂度等级
- 所需角色
- 推荐模式
- 风险等级

#### 2）团队生成
根据目标自动创建：
- Leader
- 执行 Agent
- 审核 Agent
- 裁判 Agent
- 对抗阵营 Agent

#### 3）任务拆分
自动将大目标拆分成：
- 规划任务
- 执行任务
- 验证任务
- 汇总任务

#### 4）调度策略
支持：
- 顺序调度
- 并行调度
- 竞争调度
- 追问调度
- 回合调度

#### 5）收敛策略
支持：
- 分数阈值收敛
- 时间阈值收敛
- 共识阈值收敛
- 裁判决策收敛

### 6.3 自动编排流程

1. 用户输入任务目标
2. 系统分析目标意图
3. 系统推荐最优模式
4. 系统自动组队
5. 系统自动拆分任务
6. 系统自动调度执行
7. 系统自动收敛结果并输出

### 6.4 自动编排决策建议

建议使用规则引擎 + LLM 判定的混合方案：

- **规则引擎**：处理固定映射，如“安全验证 → 红蓝攻防模式”
- **LLM 判定**：处理模糊目标，如“做一个更稳妥的发布方案”
- **人工确认**：在高风险场景下允许用户确认模式

### 6.5 自动编排的数据结构建议

```ts
interface AutoPlan {
  goal: string;
  recommendedMode: 'battle' | 'red-blue' | 'sprint' | 'turn-based' | 'review';
  agents: AgentBlueprint[];
  tasks: TaskBlueprint[];
  scheduler: {
    strategy: 'sequential' | 'parallel' | 'competitive' | 'iterative';
    convergence: 'score' | 'consensus' | 'timeout' | 'manual';
  };
}
```

### 6.6 自动编排适用场景

- 日常功能开发
- 文档生成
- 需求分析
- 方案对比
- 任务汇总与结论生成

## 7. 手动编排与自动编排的协同机制

### 7.1 双模式切换

协作页必须支持：

- **手动编排**：用户主导
- **自动编排**：系统主导
- **混合编排**：系统先给出方案，用户再调整

### 7.2 混合编排推荐流程

1. 用户输入目标
2. 系统自动生成初始方案
3. 用户在可视化编排区微调 Agent、任务和顺序
4. 用户确认启动
5. 系统执行并允许中途人工介入

### 7.3 介入点设计

混合编排必须允许用户在以下节点介入：

- 任务拆分前
- Agent 分配前
- 执行中暂停点
- 结果裁决前
- 失败恢复时

## 8. 可视化方案

### 8.1 可视化目标

最终可视化不是“装饰”，而是帮助用户理解以下五件事：

1. 谁在工作
2. 正在做什么
3. 为什么这样做
4. 结果如何变化
5. 出错后怎么恢复

### 8.2 可视化层级

#### A. 团队层
展示 Team、Leader、Teammate、Judge、Red、Blue 等角色关系。

#### B. 任务层
展示任务拆分、依赖、状态流转和当前负责 Agent。

#### C. 模式层
展示当前模式、阶段、回合和裁决结果。

#### D. 事件层
展示重要事件，如开始、分配、切换、失败、重试、恢复。

#### E. 结果层
展示最终结论、评分、对比结果、推荐方案。

### 8.3 推荐可视化组件

#### 1）团队图谱
- 节点：Team、Agent、Task、Session
- 边：指派、依赖、通信、回传
- 适合展示整体结构

#### 2）任务流时间轴
- 展示任务从创建到完成的完整链路
- 支持回放、暂停、定位异常点

#### 3）对抗竞技场
- 用于展示 5 种对抗模式
- 适合辩论、红蓝攻防、竞赛

#### 4）状态仪表盘
- 显示运行中 Agent 数量、任务进度、失败数量、恢复次数

#### 5）结果对比面板
- 支持方案评分、投票、排名和最终裁定

### 8.4 视觉表达建议

- 用颜色区分角色，不用颜色区分状态的全部语义
- 用动画强调“状态变化”，不要持续炫技
- 用线条和箭头表达关系，不用堆叠卡片表达关系
- 用时间轴表达过程，用结果卡表达结论

## 14. 实施路线图

> 说明：下面的阶段安排是面向**后期实现**的详细里程碑，不要求立即改动当前 UI 架构。执行原则仍然是“先在现有协作页上增量增强，再逐步开放高级能力”。

### 阶段 1：协作页现状梳理与能力对齐

#### 目标
在不改架构、不改配色的前提下，确认当前协作页四视图与组件职责，并把多智能体能力映射到现有结构中。

#### 当前页面结构观察
当前协作页采用四个主视图标签页，页面骨架已经稳定：

- `arena`：竞技舞台，承载主战场、对抗可视化与 Agent 列表
- `network`：协作网络，承载拓扑关系图与流向图
- `cognitive`：认知实验室，承载思维可视化与共享工作区
- `monitor`：系统监控，承载资源面板、配置台、技能插件与实时监控

同时，当前页面还保持了以下共性结构：

- 顶部品牌区与视图切换导航
- 左侧共享时间线与共享看板
- 右侧 Agent 状态与性能面板
- 底部播放控制与快捷键提示

#### 重点工作
- 梳理 `arena / network / cognitive / monitor` 的实际职责边界
- 明确每个子组件当前负责展示什么信息
- 标记哪些区域适合承载模式切换、任务流、关系图、观测流
- 对齐 WebSocket 消息类型与前端状态更新入口
- 明确哪些内容适合做“信息替换”，哪些适合做“信息增量”

#### 适配原则
- 只在现有面板中增加多智能体语义，不改页面骨架
- 保留当前标题、导航、面板位置与主色调
- 优先复用现有区域承载新能力，不新增独立 UI 容器
- 只做文案、字段和数据流增强，不做视觉体系替换

#### 产出物
- 当前页面能力映射表
- 组件职责清单
- 状态流入流出说明
- 不改架构的接入约束说明
- 现有视图与多智能体能力的映射表

#### 验收标准
- 能清楚说明现有协作页每个视图的作用
- 能明确哪些能力可直接挂载到现有组件上
- 页面结构保持不变，视觉保持不变
- 当前页面可以继续作为后续里程碑的承载底座

### 阶段 2：现有 UI 的多智能体数据增强

#### 目标
在现有 UI 中补齐多智能体数据，不改变页面骨架。

#### 重点工作
- 把 Team、Agent、Task、Session 的核心字段补齐
- 将模式状态、任务状态、Agent 状态接入现有组件
- 将事件流、回合流、任务流与现有监控区联动
- 将当前的 mock 数据逐步替换为真实编排数据
- 优先增强现有 `共享看板`、`Agent` 列表、`系统监控`、`时间线` 的内容密度

#### 产出物
- 统一数据模型
- 组件输入输出字段表
- WebSocket 消息协议草案
- 状态更新流程图
- 现有面板字段增强对照表

#### 验收标准
- 页面仍保持现有布局
- 组件内容更丰富，但没有增加理解负担
- 协作页能够显示真实编排数据而非纯 mock
- 只做增量信息注入，不改变整体配色与布局节奏

#### 本轮（M2）已执行项

### 阶段 6：cognitive / arena 编排入口与对抗状态增强

### 阶段 7：monitor 恢复入口与资源控制增强

### 阶段 8：状态模型统一与 mock 数据收敛

## 17. 结论

当前协作页要完成的不是简单 UI 重构，而是从“任务展示”跃迁为“多智能体操作系统控制台”。

最终系统应该具备四个关键词：

- **可编排**：手动和自动都能用
- **可对抗**：五种模式满足不同场景
- **可视化**：结构、过程、结果一眼能懂
- **可恢复**：异常不是终点，而是可操作状态

只要把这四个关键词做实，协作页就能真正成为当前项目的多智能体中枢。
