# 多 Agent 编排中心动态工作流详细设计文档

> 文档目标：在 `docs/multi-agent-workflow-dynamic-flow-plan.md` 的基础上，输出可直接指导开发实现的详细设计稿，重点覆盖页面结构、状态模型、交互规则、视觉规范、数据流、异常恢复，以及每个里程碑的实现列表。
>
> 适用范围：`zyfront-desktop` 多 Agent 编排中心 / 协作中心 / 工作台
>
> 设计原则：
> - 流程优先
> - 默认视口内可完成主要操作
> - 动态流转可感知
> - 详情按需展开
> - 异常可直接恢复

---

## 1. 设计目标

### 1.1 核心目标

将当前偏“信息面板式”的多 Agent 协作中心，升级为“工作流式动态编排界面”。界面应满足：

1. 用户能快速看懂当前处于哪个阶段
2. 用户能直观看到任务在 Agent 之间如何流转
3. 用户无需上下滚动即可完成主要操作
4. 用户在出现异常时能直接获得恢复动作
5. 页面视觉表达具备明确主次、层次和流转感

### 1.2 成功标准

- 核心流程在一个标准视口内完整呈现
- 任务、阶段、Agent、事件四类对象有统一关联
- 常用操作入口前置，低频信息后置
- 阻塞/失败状态可见且可操作
- 页面呈现“推进感”而不是“堆叠感”

---

## 2. 现状与设计约束

### 2.1 当前痛点

- 多 Agent、任务、事件、终端输出分散在不同区域
- 主流程不明显，用户需要自己拼上下文
- 页面纵向内容过长，依赖滚动查看
- 状态变化缺少统一反馈与动效
- 异常处理入口不够集中

### 2.2 设计约束

- 不能牺牲现有能力：spawn、stop、kill、会话恢复、事件时间线仍需保留
- 需要兼容多种运行模式（如 tmux / in-process / backend blocking）
- 需要为后续扩展保留状态模型与布局弹性
- 样式增强不能增加信息噪声

---

## 3. 总体架构设计

### 3.1 设计分层

页面拆分为四层：

1. **流程阶段层**：表达整体进度
2. **工作流主视图层**：表达流转路径
3. **右侧上下文层**：表达选中对象详情与操作
4. **底部事件层**：表达最近变化与恢复反馈

### 3.2 设计原则映射

| 设计原则 | 对应实现 |
|---|---|
| 流程优先 | 顶部阶段条 + 中央工作流图 |
| 默认视口内操作 | 固定高度布局 + 折叠/抽屉详情 |
| 动态流转可感知 | 节点高亮、连线流光、状态切换动画 |
| 详情按需展开 | 侧栏、弹层、hover 提示 |
| 异常可恢复 | 节点内嵌恢复按钮 + 错误摘要 |

## 4. 页面信息架构设计

### 4.1 页面布局

建议使用“顶部 + 中央双栏 + 底部”的固定视口结构：

- **顶部区域**
  - 标题
  - 当前运行模式
  - 总体状态
  - 阶段进度条
  - 全局快捷操作

- **中央主区域**
  - 左侧：工作流画布
  - 右侧：当前节点详情面板

- **底部区域**
  - 最近事件
  - 错误/恢复提示
  - 快捷命令

### 4.2 信息优先级

#### 第一优先级
- 当前流程阶段
- 当前正在执行的节点
- 当前阻塞/异常状态
- 主要 CTA

#### 第二优先级
- 当前任务摘要
- Agent 状态
- 最近事件
- 最近回传结果

#### 第三优先级
- 详细日志
- 历史记录
- 调试信息
- 低频配置项

### 4.3 布局行为

- 页面默认不依赖纵向滚动即可完成核心操作
- 低频信息通过折叠、浮层或抽屉展示
- 若内容过多，优先横向分区而不是继续纵向堆叠

## 5. 核心数据模型设计

### 5.1 核心对象

#### WorkflowStage
表示整个协作流程所处的阶段。

字段建议：
- `id`
- `name`
- `status`：`idle | active | done | blocked | error`
- `order`
- `description`
- `nextHint`

#### WorkflowNode
表示工作流中的节点，可以是任务、Agent、恢复节点。

字段建议：
- `id`
- `type`：`control | task | agent | recovery`
- `title`
- `status`：`pending | active | done | blocked | error | retrying`
- `ownerId`
- `summary`
- `progress`
- `metrics`
- `actions`

#### WorkflowEdge
表示节点之间的流转关系。

字段建议：
- `from`
- `to`
- `status`：`idle | active | done | blocked`
- `label`
- `direction`

#### AgentState
表示 Agent 当前运行状态。

字段建议：
- `agentId`
- `role`
- `status`
- `currentTaskId`
- `sessionId`
- `mode`
- `lastSeenAt`
- `health`

#### TaskState
表示任务卡状态。

字段建议：
- `taskId`
- `title`
- `status`
- `stageId`
- `assigneeId`
- `priority`
- `resultSummary`
- `blockReason`

#### EventItem
表示事件与流程变化。

字段建议：
- `id`
- `type`
- `severity`
- `message`
- `sourceId`
- `relatedNodeId`
- `timestamp`
- `actions`

### 5.2 状态驱动关系

- `WorkflowStage` 决定页面顶部阶段条
- `WorkflowNode` 决定中央工作流画布
- `AgentState` 决定节点颜色、标签与可操作项
- `TaskState` 决定任务卡流转与摘要
- `EventItem` 决定底部事件条与异常提示

### 5.3 状态同步规则

当任意一个对象变化时，需同步影响：
- 当前高亮节点
- 阶段条状态
- 右侧详情内容
- 底部最近事件
- 可用操作按钮

## 6. 核心交互设计

### 6.1 页面进入

#### 用户看到的内容
- 当前阶段
- 当前活跃节点
- 最近 1~3 条关键事件
- 主要操作按钮

#### 设计要求
- 首屏即解释“现在正在发生什么”
- 如果没有任务，则显示引导创建入口
- 如果有恢复点，则优先显示恢复提示

### 6.2 阶段切换

- 点击阶段可跳转对应阶段视图
- 当前阶段高亮
- 已完成阶段支持回看
- 被阻塞阶段显示恢复入口

### 6.3 节点选择

- 点击节点后，右侧详情面板展示该节点信息
- 节点被选中时，画布中其边框/发光高亮
- 与之相关的上下游连线同步高亮

### 6.4 任务流转

任务卡在不同节点间流转时，需要体现：
- 移动方向
- 当前负责人变化
- 状态变化
- 预计完成时间或耗时

### 6.5 恢复操作

恢复动作必须前置在节点附近，避免用户寻找入口。

可恢复动作包括：
- 重试
- 重新分配
- 跳过
- 停止当前节点
- 重新连接会话
- 查看错误详情

---

## 7. 视觉设计规范

### 7.1 视觉风格

建议采用深色工作台风格：
- 背景层偏暗
- 卡片层适度抬起
- 节点状态用高辨识度颜色表达
- 动效要轻，避免干扰内容阅读

### 7.2 状态颜色建议

- `idle`：灰蓝
- `active`：青蓝 / 亮蓝
- `done`：绿色
- `blocked`：橙色
- `error`：红色
- `retrying`：紫蓝或渐变高亮

### 7.3 关键组件样式

#### 阶段条
- 横向展示
- 当前阶段突出
- 已完成阶段低饱和
- 阻塞阶段带告警符号

#### 节点卡片
- 圆角、轻阴影、细边框
- 节点标题、状态、摘要、负责人信息清晰分区
- 当前节点加入发光或边框脉冲

#### 连线
- 已完成连线低亮
- 当前流转连线使用渐变流动效果
- 阻塞链路用警示色打断

#### 详情面板
- 标题区 + 信息分组 + 操作区
- 文本可折叠
- 关键数据加粗或标签化

### 7.4 动效原则

- 只给“状态变化”加动效，不给静态信息加噪声
- 动效持续时间短，反馈明确
- 以高亮、流动、切换、进场为主
- 避免大面积抖动和过度炫技

### 8.1 流程状态机

建议统一为：
- `not_started`
- `preparing`
- `orchestrating`
- `executing`
- `aggregating`
- `completed`
- `blocked`
- `error`

### 8.2 任务状态机

建议统一为：
- `draft`
- `queued`
- `assigned`
- `running`
- `waiting_review`
- `done`
- `failed`
- `retrying`

### 8.3 Agent 状态机

建议统一为：
- `idle`
- `busy`
- `waiting_input`
- `connecting`
- `degraded`
- `offline`

### 8.4 恢复状态机

建议统一为：
- `available`
- `recovering`
- `recovered`
- `failed`

# M1 交互收敛与反馈统一

## 9.1 目标

让用户“看得懂、点得对、知道发生了什么”，为后续工作流化做基础。

## 9.2 实现列表

### 9.2.1 统一全局反馈层
- 增加统一 toast / banner / inline feedback 体系
- 所有关键动作返回统一成功/失败提示
- 反馈文案结构固定为：原因 + 结果 + 建议动作
- 支持成功、失败、警告、恢复成功四种类型

### 9.2.2 统一关键动作结果
- 统一 `Spawn`、`Stop`、`Kill`、`Send` 的结果展示格式
- 成功时明确确认对象与时间
- 失败时显示错误来源、影响范围和下一步建议

### 9.2.3 顶部增加“下一步建议”区域
- 根据当前流程状态给出推荐动作
- 例如：创建任务、指派 agent、查看阻塞、重新恢复
- 支持自动根据状态刷新建议内容

### 9.2.4 事件分层显示
- 将事件分为业务事件与技术事件
- 默认显示业务事件
- 技术事件折叠到二级视图

### 9.2.5 新手引导空态
- 无任务状态时展示引导卡
- 引导用户创建任务或恢复会话
- 清晰说明下一步入口

### 9.2.6 首次任务闭环提示
- 当任务首次完成时给出阶段完成反馈
- 展示回传摘要与后续操作建议

## 9.3 验收标准
- 关键动作反馈覆盖率 100%
- 用户不会因静默失败而误判成功
- 页面入口的含义更清晰

---

# M2 任务导向协作流

## 9.4 目标

把多 Agent 协作从“按钮驱动”升级为“任务驱动”。

## 9.5 实现列表

### 9.5.1 引入任务卡模型
- 新增任务卡 UI
- 每张任务卡包含目标、负责人、状态、进度、摘要
- 卡片支持拖拽/切换/选中

### 9.5.2 任务分配能力
- 任务可直接指派到某个 Agent
- 分配后自动更新任务状态和流程节点
- 支持重新分配

### 9.5.3 任务阶段展示
- 任务具备自己的子阶段展示
- 显示“待分配 / 执行中 / 等待回传 / 已完成”
- 阶段变化在卡片上实时体现

### 9.5.4 任务汇总视图
- 汇总每个任务的最新结论
- 展示阻塞点、证据、下一步
- 支持快速进入对应 Agent 详情

### 9.5.5 任务与 Agent 联动
- 选中任务时高亮对应 Agent
- 选中 Agent 时显示其负责的任务
- 保持双向关联一致

## 9.6 验收标准
- 用户可通过任务理解协作，而不是通过技术对象理解协作
- 页面中任务信息比原始事件更具主导性
- 单任务场景下用户切页次数明显下降

---

# M3 可靠性与恢复

## 9.7 目标

解决“能跑”到“敢用”的差距，把异常恢复产品化。

## 9.8 实现列表

### 9.8.1 恢复中心入口
- 页面顶部或右侧增加恢复中心入口
- 聚合所有可恢复项
- 显示恢复建议优先级

### 9.8.2 阻塞节点处理
- 阻塞节点上直接显示恢复按钮
- 提供重试、重新分配、查看详情、停止等操作
- 阻塞原因必须结构化

### 9.8.3 会话恢复机制
- 页面重开时自动检查未完成会话
- 支持按任务恢复、按 Agent 恢复
- 恢复成功后回填流程状态

### 9.8.4 无响应处理
- Agent 无响应时状态进入 `degraded`
- 提示用户是否重连、重试或切换执行方式
- 记录失败次数与最后一次可用时间

### 9.8.5 异常重试策略
- 对可重试错误增加统一重试按钮
- 连续失败时提示升级处理建议
- 防止重复点击导致状态错乱

### 9.8.6 可观测性补充
- 展示失败率、恢复次数、阻塞时长等关键指标
- 提供最近异常摘要

## 9.9 验收标准
- 会话恢复逻辑稳定
- 阻塞状态可直接从节点处理
- 异常后用户能快速回到可操作状态

---

# M4 高级体验与治理

## 9.10 目标

支持团队规模使用，并增强安全性、资源治理与协作质量。

## 9.11 实现列表

### 9.11.1 权限与危险操作确认
- 对 Kill、批量操作、强制重连增加确认
- 高风险行为需要明确说明影响
- 可配置二次确认文案

### 9.11.2 并发与资源治理
- 设定最大并发 Agent 数
- 任务过载时提供排队提示
- 支持自动降载和优先级调度

### 9.11.3 协作质量标准化
- Teammate 回传摘要统一模板
- 至少包含：结论、证据、风险、下一步
- 让 leader 汇总更稳定

### 9.11.4 团队态势视图
- 显示当前活跃 Agent、阻塞数、完成数、异常数
- 给出整体健康度
- 支持快速定位异常来源

### 9.11.5 审计与追踪
- 记录关键操作链路
- 便于回放和排查
- 支持导出摘要

## 9.12 验收标准
- 团队级使用具备安全边界
- 危险操作可追踪可确认
- 页面从个人工具进一步进化为团队协作台

---

# M5 样式统一与动效增强

## 9.13 目标

让页面真正具备“工作流流转”的视觉感受，同时保持克制。

## 9.14 实现列表

### 9.14.1 统一视觉规范
- 统一卡片、标题、标签、按钮、间距样式
- 统一状态色板
- 统一分组标题层级

### 9.14.2 连线流动效果
- 当前流转线加入轻量流动渐变
- 完成线与未完成线清晰区分
- 阻塞时中断视觉表达

### 9.14.3 节点切换动效
- 选择节点时有平滑高亮过渡
- 状态变化时有颜色渐变或边框动画
- 保持节奏轻快

### 9.14.4 详情面板过渡
- 选中对象切换时平滑更新内容
- 避免页面跳动
- 长文本折叠后过渡自然

### 9.14.5 空态与异常态美化
- 空态更像任务引导，而不是空白页
- 异常态强调修复路径而不是仅提示失败

## 9.15 验收标准
- 页面主次信息清晰
- 动效增强理解而非干扰阅读
- 页面有明显的“流转感”

## 10. 技术实现建议

### 10.1 状态管理

建议将工作流状态独立为统一状态域，至少包含：
- 流程阶段
- 工作流节点
- 选中节点
- 任务列表
- Agent 列表
- 最近事件
- 恢复动作

### 10.2 更新原则

- 任何事件变更必须通过统一状态入口更新
- 节点状态与 Agent 状态不得各自独立漂移
- 右侧详情和底部事件必须由当前状态派生

### 10.3 组件拆分建议

- `WorkflowStageBar`
- `WorkflowCanvas`
- `WorkflowNodeCard`
- `WorkflowDetailPanel`
- `WorkflowEventRail`
- `RecoveryActionPanel`
- `GlobalFeedbackCenter`

### 10.4 数据适配层

建议增加一层适配器，将后端事件转换为页面可用模型：
- 原始事件 → 业务事件
- Agent 状态 → 节点状态
- 任务执行 → 流转进度
- 错误信息 → 恢复卡片

## 12. 测试与验收建议

### 12.1 功能测试
- 阶段切换是否正确
- 节点选择是否联动右侧面板
- 任务分配是否更新流转状态
- 恢复动作是否可执行

### 12.2 视觉测试
- 深色主题下对比度是否足够
- 状态颜色是否区分明显
- 小屏下是否仍能保持主线清晰

### 12.3 交互测试
- 核心操作是否在默认视口内完成
- 是否需要滚动才能找到关键按钮
- 异常态是否能直接恢复

### 12.4 稳定性测试
- 高频事件更新是否导致界面抖动
- 多 Agent 并发状态变化是否同步
- 恢复与重试是否能正确回填状态

## 13. 推荐的交付顺序

1. 先做布局重构，确定固定视口结构
2. 再做流程阶段条和工作流主视图
3. 再接入任务卡与 Agent 联动
4. 再补恢复中心与异常节点处理
5. 最后统一样式、动效、埋点和治理能力

---

## 14. 最终交付形态

完成后页面应具备以下体验：

- 用户一进来就知道现在到哪一步了
- 用户看得到任务如何在 Agent 间流转
- 用户不用上下滚动也能完成大部分操作
- 出现异常时可以直接在节点上恢复
- 页面既有工作流感，也有可观测性和治理能力
