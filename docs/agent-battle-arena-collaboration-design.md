## 1. 文档目标与范围

### 1.1 文档目标
本文档用于定义 **Agent Battle Arena 多智能体协作页** 的完整设计规范。该页面的目标不是简单展示 Agent 列表，而是将多智能体协作过程构建成一个**像素街机风格的实时战斗/协作舞台**，让用户能够：

- 直观理解协作过程
- 实时观察 Agent 状态与关系
- 控制协作节奏
- 干预协作行为
- 回放历史过程
- 导出结果与战报
- 查看系统资源与运行状况

### 1.2 文档范围
本文档覆盖以下内容：

- 页面定位与产品目标
- 视觉风格与布局规范
- 组件设计与模块职责
- 动画、特效与反馈设计
- 交互模式与快捷键设计
- 状态与数据流设计
- 系统监控与可观测性设计
- 实施里程碑与验收标准

### 1.3 不在范围内的内容
本文档不涉及：

- 后端 Agent 核心算法实现细节
- LLM Prompt 细节
- 具体数据库表结构
- 某一特定前端框架下的代码实现
- 当前已有实现的兼容修正策略

## 2. 产品定位与页面愿景

### 2.1 页面定位
多智能体协作页是整个系统的**协作控制中心**，也是 Agent Battle Arena 的“主舞台”。它同时承担四种角色：

1. **舞台**
   - 用于展示多 Agent 的协作、辩论、交锋、任务流转。

2. **控制台**
   - 用于暂停、播放、加速、回放、聚焦、干预。

3. **观测台**
   - 用于查看状态、关系、时间轴、数据流和监控指标。

4. **战报台**
   - 用于回顾历史、导出结果、查看总结与关键事件。

### 2.2 页面愿景
页面希望达到的体验是：

- 用户进入页面后，会感觉像进入一个复古街机战场
- 每个 Agent 都是一个有性格、有状态、有能量、有输出的角色
- 协作过程不是抽象状态切换，而是可视化的“战斗 / 交接 / 合作 / 竞争”
- 所有重要动作都有即时反馈
- 系统状态、流程状态与视觉状态高度一致
- 整体风格统一为 **8-bit 像素艺术 + 街机游戏 + 赛博朋克**

### 2.3 目标用户
- AI 系统开发者
- 多智能体系统研究人员
- 技术产品经理
- 交互设计师
- 前端工程师
- 对 AI 协作过程有强观察需求的用户

## 3. 核心设计原则

### 3.1 视觉原则

#### 3.1.1 像素风原则
- 使用 8-bit / 16-bit 像素艺术语言
- 所有图像、图标、角色尽量保持像素化表现
- 关键元素使用硬边框、强轮廓、高对比色

#### 3.1.2 街机原则
- 界面像街机游戏控制台
- 核心信息像游戏 HUD
- 交互像“投币开始 / 选角色 / 发起战斗 / 暂停回放”

#### 3.1.3 赛博朋克原则
- 背景以深空黑、暗夜蓝为主
- 霓虹色做强调
- 发光、粒子、扫描线等用于增强科技感
- 强调“数字竞技场”而不是普通后台

### 3.2 交互原则
- 所有操作必须即时响应
- 所有关键状态必须可感知
- 所有高频操作必须一键可达
- 所有低频操作必须收口
- 允许用户随时暂停、回放、聚焦与干预
- 支持鼠标、键盘、触控多输入方式

### 3.3 信息组织原则
- 主舞台优先
- 状态辅助
- 控制明确
- 监控收口
- 历史可回放
- 数据可分析

### 3.4 性能原则
- 目标 60fps
- 动画尽量使用 Canvas / PixiJS / CSS 动画组合
- 复杂图形按需渲染
- 非当前内容可折叠或延迟加载
- 粒子系统采用对象池

## 5. 页面总体布局设计

### 5.1 总体布局结构
页面建议采用如下布局：

1. **顶部 Header 区**
2. **中部主舞台区**
3. **左侧 Agent 与关系区**
4. **右侧任务与详情区**
5. **底部控制与监控区**
6. **可选回放浮层 / 详情抽屉**

这一结构与设计文档中的“主舞台 + 数据面板 + 控制面板 + 监控面板”一致。

### 5.2 顶部 Header 区

#### 5.2.1 功能
- 展示系统名称
- 展示当前协作模式
- 展示当前议题
- 展示时间 / 回合 / 状态
- 提供全局设置入口

#### 5.2.2 内容建议
- 左侧：`AGENT BATTLE ARENA`
- 中间：当前模式、当前主题、当前状态
- 右侧：计时器、暂停状态、设置按钮、导出按钮

#### 5.2.3 风格要求
- 80px 左右高度
- 像素边框
- 轻微发光
- 标题高亮
- 信息区分组明显

### 5.3 中部主舞台区

#### 5.3.1 功能
主舞台是页面视觉与逻辑重心，用于表现：

- Agent 对抗
- Agent 协作
- 任务交接
- 论点流动
- 数据碰撞
- 结果涌现

#### 5.3.2 视觉内容
- 中央舞台背景
- 关键 Agent 动画
- 论点/任务弹幕
- 光束、爆炸、交锋特效
- 回合提示
- 当前阶段提示

#### 5.3.3 场景语义
不同协作模式下，主舞台会承担不同语义：

- 对抗：战斗交锋
- 协作：共同构建
- 流水线：阶段传递
- 脑暴：观点涌现
- 竞赛：排名争夺

### 5.4 左侧 Agent 与关系区

#### 5.4.1 功能
展示 Agent 个体与群体关系。

#### 5.4.2 包含内容
- Agent 节点列表或弧形集群
- 当前状态
- HP / 能量 / 得分
- 当前行为
- 协作网络图

#### 5.4.3 设计目标
让用户看懂：
- 谁是核心节点
- 谁处于活跃状态
- 谁正在等待
- 谁与谁有强关联

### 5.5 右侧任务与详情区

#### 5.5.1 功能
用于显示当前选中对象的完整上下文。

#### 5.5.2 显示内容
- 当前任务
- 当前 Agent 详情
- 当前关系链
- 当前回传内容
- 当前事件说明
- 可用操作

#### 5.5.3 设计目标
把“选中某个节点后该看什么”变得非常清晰。

### 5.6 底部控制与监控区

#### 5.6.1 控制区内容
- 播放 / 暂停 / 继续
- 速度切换
- 回放
- 聚焦 Agent
- 视角切换
- 手动干预
- 导出

#### 5.6.2 监控区内容
- 时间轴
- 事件流
- 性能监控
- Token / 成本趋势
- 告警信息
- 记录导出

## 7. 协作模式设计

页面必须支持多种协作模式，并且每种模式有不同的视觉语义与数据组织方式。

### 7.1 对抗模式
#### 目标
让多个 Agent 针对同一问题展开观点冲突和方案博弈。

#### 表现
- 双方阵营明显
- 光束对冲
- 爆炸反馈
- 胜负趋势明显

### 7.2 协作模式
#### 目标
让多个 Agent 共同完成一个任务。

#### 表现
- 共同推进进度
- 节点之间连线更密
- 舞台更强调“合成”与“构建”

### 7.3 流水线模式
#### 目标
将复杂任务拆分为多个阶段，由不同 Agent 接力完成。

#### 表现
- Agent 按顺序激活
- 上游输出成为下游输入
- 强调交接、处理、汇总

### 7.4 脑暴模式
#### 目标
让多个 Agent 同时生成观点，最后聚合。

#### 表现
- 观点同时涌现
- 大量弹幕式内容
- 最终收束为结论

### 7.5 竞赛模式
#### 目标
让多个 Agent 并行竞争同一目标。

#### 表现
- 排名实时变化
- 得分浮动
- 冲刺、反超、落后等状态明显

## 9. 交互与控制设计

### 9.1 交互层级

#### 基础控制
- 播放
- 暂停
- 继续
- 切速
- 前进 / 后退

#### 导航控制
- 聚焦 Agent
- 切换视角
- 定位事件节点

#### 高级操作
- 手动干预
- 能量注入
- 召唤援军
- 重置当前回合

#### 系统功能
- 截图保存
- 导出结果
- 全屏
- 回放

### 9.2 速度控制
推荐固定档位：

- `1x` 正常模式
- `2x` 快速模式
- `4x` 专家模式
- `8x` 极速模式

高速模式应伴随明显警告。

### 9.3 键盘快捷键
沿用街机 / MAME 风格设计：

- `Space`：暂停 / 继续
- `Tab`：切换速度
- `F`：全屏
- `P`：截图保存
- `R`：回放
- `← / →`：前进 / 后退
- `1-6`：快速聚焦 Agent
- `Ctrl+S`：导出记录

### 9.4 手动干预
#### 支持能力
- 能量注入
- 临时无敌
- 召唤辅助 Agent
- 重置当前协作阶段

#### 设计要求
- 需二次确认
- 有危险提示
- 执行后必须有明显反馈

## 11. 可观测性与监控设计

### 11.1 监控目标
协作页不仅要展示结果，还要能分析过程。

### 11.2 监控内容
- Agent 在线状态
- Agent 负载
- 消息流
- 任务吞吐
- 回合耗时
- Token 消耗
- 资源利用率

- Agent 离线
- 任务卡住
- 协作链路断裂
- 资源占用过高
- 回合长时间无进展
- 结果异常

### 11.4 监控呈现方式
- 顶部摘要
- 底部趋势图
- 右侧告警卡
- 可折叠的调试抽屉
- 事件时间轴联动

## 13. 无障碍与可用性设计

尽管页面具有较强风格化，但仍需满足基本可用性：

- 高对比文本

- 按钮有焦点态
- 支持键盘控制
- 动画可关闭
- 读屏可理解关键状态
- 危险操作有确认步骤

## 17. 风险与应对策略

### 17.1 风险：信息过载
#### 说明
页面功能很多，容易变成信息堆叠。

#### 应对
- 分区收口
- 分层显示
- 当前模式优先
- 低频内容折叠

### 17.2 风险：动画影响性能
#### 说明
大量粒子、发光与图谱可能影响帧率。

#### 应对
- 对象池
- 降频
- 懒加载
- 按需渲染

### 17.3 风险：风格化过强影响可读性
#### 说明
像素风、CRT、霓虹效果可能影响信息阅读。

#### 应对
- 保持强对比文本
- 控制装饰强度
- 关键状态永远清晰优先

### 17.4 风险：交互路径过多
#### 说明
控制项太多时用户会不知道先做什么。

#### 应对
- 主流程唯一
- 高亮推荐动作
- 阻塞时优先恢复路径
- 低频按钮收口

## 18. 新页面结构映射

本节将上述设计要求映射为**全新的协作页结构**。该结构不以当前实现为前提，而是严格按照设计文档定义页面应有形态。

### 18.1 页面结构总览
建议的新页面采用“**一条主线、四个节点、三层信息、两级操作**”的结构：

1. **顶部全局栏（Global Header）**
   - 系统标题
   - 当前模式
   - 当前主题
   - 时间 / 回合 / 运行状态
   - 全局设置与导出入口

2. **协作流程导航条（Flow Navigator）**
   - 创建
   - 补齐
   - 分派
   - 观察
   - 当前步骤高亮
   - 已完成步骤回看

3. **主舞台区（Battle Stage）**
   - 中央协作/战斗表现区
   - Agent 交锋、任务传递、观点涌现
   - 舞台特效与结果反馈

4. **左侧信息区（Agent & Network）**
   - Agent 节点集群
   - 角色状态
   - 协作关系图
   - 当前活跃节点

5. **右侧详情区（Task & Detail）**
   - 当前选中任务
   - Agent 详情
   - 回传内容
   - 可执行操作
   - 推荐动作

6. **底部观测与控制区（Control & Monitor）**
   - 播放 / 暂停 / 继续
   - 速度切换
   - 回放
   - 聚焦
   - 时间轴
   - 事件流
   - 系统监控

### 18.2 分层展示规则
新页面应按照以下规则组织信息：

#### 一级层：主线层
始终只强调当前阶段最关键的 1 个动作与 1 个状态：
- 当前处于哪个步骤
- 当前最推荐用户做什么

#### 二级层：工作层
用于展示协作过程中的核心对象：
- Agent
- 任务
- 关系
- 进度
- 回传

#### 三级层：观测层
用于展示过程历史与系统状态：
- 时间轴
- 日志
- 告警
- 监控指标
- 回放记录

### 18.3 各区域与设计文档的映射关系

#### 18.3.1 顶部全局栏
对应设计文档中的：
- 项目概述与视觉风格
- 交互设计中的全局控制面板

#### 18.3.2 流程导航条
对应设计文档中的：
- 页面信息架构中的主线定义
- 里程碑文档中的流程推进思路

#### 18.3.3 主舞台区
对应设计文档中的：
- 布局与组件设计中的 Battle Stage
- 动画与特效规范中的交锋与粒子系统

#### 18.3.4 左侧 Agent & Network 区
对应设计文档中的：
- AgentNode 组件
- 协作关系网络图
- 工作流时间轴中的主体对象

#### 18.3.5 右侧详情区
对应设计文档中的：
- Agent 状态卡
- 任务详情卡
- 回传内容卡
- 可执行操作面板

#### 18.3.6 底部观测与控制区
对应设计文档中的：
- 控制面板
- 监控面板
- 系统架构与可观测性设计

### 18.4 新页面的推荐交互流
新页面的理想交互路径如下：

1. 用户进入页面，查看全局状态与当前模式
2. 通过流程导航确认当前阶段
3. 在主舞台中观察 Agent 协作或交锋
4. 在左侧查看 Agent 与网络结构
5. 在右侧查看当前选中对象详情
6. 在底部进行播放、回放、干预与监控
7. 当发生异常时，恢复与告警入口自动提升优先级

### 18.5 新页面的视觉主次关系
- **主视觉**：中央战斗舞台
- **次视觉**：Agent 集群与关系图
- **辅助视觉**：右侧详情与底部控制
- **低优先级视觉**：历史日志、监控趋势、调试信息

### 18.6 与原型风格的对应要求
新页面不仅要符合协作逻辑，还要继承原型中的视觉语言：
- 深色背景与霓虹强调
- 像素风角色与像素 UI
- 竞技场 HUD 风格
- 复古街机控制台感
- 轻微扫描线、闪烁与粒子装饰

### 18.7 新页面的结构总结
新页面不是一个“功能列表页”，而是一个**按流程组织、按状态驱动、按舞台呈现**的多智能体协作战场。其结构必须满足：

- 用户一眼能看懂下一步
- 过程一眼能看懂谁在做什么
- 结果一眼能看懂发生了什么
- 监控一眼能看懂系统是否健康

## 19. 总结

多智能体协作页的核心不是“功能展示”，而是把多 Agent 协作过程设计成一个**具有街机感、战斗感、实时感和可观测性**的完整舞台。

本设计文档定义了：

- 页面定位
- 视觉体系
- 布局结构
- 核心组件
- 动画特效
- 交互控制
- 数据状态
- 监控体系
- 响应式策略
- 里程碑推进方案
- 新页面结构映射

最终目标是打造一个真正符合 **Agent Battle Arena** 概念的协作中枢：

- 能看
- 能控
- 能回放
- 能分析
- 能干预
- 能导出
- 而且足够像一个“像素大战竞技场”
