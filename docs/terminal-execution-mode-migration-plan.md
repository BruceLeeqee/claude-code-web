# 多 Agent 任务分配与主 Agent 模型重构计划

## 1. 背景与问题定义

当前 `workbench` 的多 Agent 面板已经具备基础的执行模式切换、tmux / in-process / iTerm2 后端、Agent 创建与事件时间线能力，但产品语义仍然混杂，主要存在以下问题：

1. **执行模式与任务分配混为一体**
   - 页面中的下拉框当前承担的是后端执行模式切换（`auto / in-process / tmux / iterm2`）。
   - 用户容易误解为“任务分配对象”或“负责人切换”。
   - 实际上这两个概念应该分离：
     - 执行模式决定 Agent 如何运行。
     - 任务分配决定任务由谁负责。

2. **任务卡语义不清晰**
   - 当前任务区域更像“摘要展示”或“静态说明”。
   - 任务内容不可明显编辑，也没有强制绑定负责人或目标 Agent。
   - 用户无法明确知道一个任务是发给主 Agent、子 Agent，还是仅用于内部记录。

3. **主 Agent 概念没有显式建模**
   - 页面中存在“主终端 / lead agent”的隐含语义，但没有形成清晰的产品层概念。
   - 用户看不出谁是调度中心、谁负责接收用户输入、谁负责拆分任务。

4. **Agent 与任务之间缺少绑定关系**
   - 目前更偏向于“创建一个 Agent，再通过事件/消息观察其行为”。
   - 但对于实际使用，用户需要的是：
     - 任务可指定负责人
     - 任务可指定给某个 Agent
     - 任务状态可追踪
     - 任务结果可回流到主 Agent

5. **UI 信息层次不够清晰**
   - 面板中有执行模式、Agent 列表、事件时间线、恢复中心、可观测性等多个区块。
   - 但缺少明确的主次关系，导致用户不知道应先看什么、先操作什么。

## 2. 重构目标

本次重构的目标不是单纯加按钮，而是建立一套更清晰的多 Agent 工作模型：

### 2.1 语义目标
- 将 **执行模式** 与 **任务分配** 完全分离。
- 将 **主 Agent（Lead Agent）** 显式展示为调度中心。
- 将 **子 Agent（Teammate）** 视为可分配任务的执行者。
- 将 **任务卡** 定义为可编辑、可指派、可追踪的工作单元。

### 2.2 体验目标
- 用户能在 30 秒内理解：
  1. 当前系统使用什么执行模式。
  2. 谁是主 Agent。
  3. 任务如何创建、编辑、分配。
  4. 子 Agent 当前在做什么。
- 用户能直接指定某个 Agent 接单。
- 用户能在事件时间线中回看任务流转与状态变化。

### 2.3 技术目标
- 保持现有多 Agent 后端能力不被破坏。
- 保持 tmux / WSL2 / in-process / iTerm2 的执行逻辑可用。
- 尽量复用现有 `MultiAgentOrchestratorService`、`WorkbenchTeamVm`、事件总线、恢复快照等基础设施。
- 将新增的“任务分配”与“主 Agent”建模为明确的数据结构，而不是仅存在于 UI 文案中。

## 3. 当前状态梳理

### 3.1 已有能力
- 支持执行模式：`auto / in-process / tmux / iterm2`
- 支持 Agent 创建、停止、强制终止、附着/分离
- 支持事件时间线展示
- 支持恢复中心与可观测性展示
- 支持 tmux 在 Windows 下通过 WSL2 执行

### 3.2 现存问题
- “模式切换”与“任务分配”没有区分。
- 任务实体不够明确。
- 主 Agent 概念未显示。
- Agent 卡片中缺少“负责人 / 目标 / 当前任务”的结构化信息。
- 右侧面板信息密度高，但优先级不明确。

## 4. 建议的新产品模型

### 4.1 核心角色定义

#### 主 Agent（Lead Agent）
- 系统中的调度中心。
- 默认接收用户输入。
- 负责拆分任务、分派子任务、汇总结果。
- 在 UI 中应始终可见。

#### 子 Agent（Teammate）
- 负责执行局部任务。
- 可被主 Agent 指派。
- 可被用户直接选中查看、停止、附着、分离。

#### 任务（Task）
- 一个明确的工作单元。
- 具备标题、描述、状态、优先级、负责人、目标 Agent、创建时间、更新时间等字段。
- 支持编辑与重分配。

### 4.2 任务状态建议
- `draft`：草稿，尚未分配。
- `queued`：已排队等待执行。
- `assigned`：已分配给某个 Agent。
- `running`：正在执行。
- `blocked`：被阻塞，需要人工干预。
- `done`：已完成。
- `failed`：执行失败。
- `cancelled`：已取消。

### 4.3 任务分配方式
- 默认分配给主 Agent。
- 可显式指定到某个子 Agent。
- 可支持“未分配池”，由主 Agent 后续派发。
- 可支持快速切换负责人。

## 5. UI 重构建议

### 5.1 页面结构建议
建议将右侧多 Agent 面板拆成更明确的层级：

1. **执行模式区**
   - 仅负责切换 `auto / in-process / tmux / iterm2`
   - 不承载任务分配语义

2. **主 Agent 区**
   - 显式展示 Lead Agent
   - 展示当前职责、状态、最后更新时间
   - 提供“发送给主 Agent”的快捷输入

3. **任务卡区**
   - 可编辑任务标题/内容
   - 可指定负责人
   - 可指定目标 Agent
   - 支持状态切换

4. **子 Agent 列表区**
   - 展示已创建的子 Agent
   - 每个卡片显示：当前状态、连接态、恢复态、任务数、最近事件
   - 支持附着/分离/停止/强制终止

5. **事件时间线区**
   - 仅做可追踪的事件回放
   - 点击事件可跳转到对应 Agent 或任务

### 5.2 控件语义建议
- 下拉框：只切执行模式。
- `Spawn`：创建新 Agent，不等于分配任务。
- `Assign`：把任务分给指定 Agent。
- `Attach / Detach`：控制 tmux 会话连接状态。
- `Stop / Kill`：终止运行。

### 5.3 视觉建议
- 主 Agent 使用更醒目的样式。
- 任务卡与 Agent 卡分层展示，避免混淆。
- 状态徽标采用统一颜色体系：
  - 运行中：蓝/绿
  - 阻塞：黄
  - 错误：红
  - 草稿/未分配：灰
- 任务分配字段和 Agent 状态字段应在视觉上分区。

## 6. 数据结构建议

### 6.1 新增任务模型
建议新增类似以下结构：

- `TaskItem`
  - `id`
  - `title`
  - `description`
  - `status`
  - `priority`
  - `assigneeAgentId?`
  - `createdAt`
  - `updatedAt`
  - `source`
  - `tags?`

### 6.2 任务与 Agent 的关联
- `WorkbenchTeamVm` 中可扩展出：
  - `leadAgentId`
  - `tasks`
  - `unassignedTasks`
  - `assignedTaskCount`
- `WorkbenchTeammateVm` 中可扩展出：
  - `currentTaskId?`
  - `taskCount`
  - `canAcceptTask`

### 6.3 事件模型建议
建议新增或细化事件：
- `task.created`
- `task.updated`
- `task.assigned`
- `task.started`
- `task.blocked`
- `task.done`
- `task.failed`
- `task.cancelled`
- `agent.promoted` / `agent.demoted`（如需要）

## 7. 分阶段实施计划

### Phase 0：概念澄清
目标：先让 UI 和数据模型在语义上对齐。

交付内容：
- 执行模式与任务分配职责分离。
- 主 Agent 概念显式展示。
- 任务卡命名、状态、负责人统一。

验收标准：
- 用户能清楚分辨“切模式”和“分任务”是两件事。
- UI 上能看到主 Agent。

### Phase 1：任务实体落地
目标：建立可编辑任务模型。

交付内容：
- 新增任务数据结构。
- 支持任务创建、编辑、删除、状态更新。
- 任务可指定 assignee。

验收标准：
- 可创建任务卡。
- 任务内容可修改并持久化。
- 任务有明确负责人。

### Phase 2：Agent 任务绑定
目标：任务与 Agent 建立可观测的关联。

交付内容：
- Agent 卡显示任务数/当前任务。
- 任务可分配给某个子 Agent。
- 事件时间线记录任务流转。

验收标准：
- 分配后 Agent 卡能即时反映任务变化。
- 事件时间线能回看任务分配过程。

### Phase 3：主 Agent 调度体验
目标：让主 Agent 成为显式调度中心。

交付内容：
- 主 Agent 置顶展示。
- 默认输入发送给主 Agent。
- 支持从主 Agent 快速派发到子 Agent。

验收标准：
- 用户能一眼看出谁在调度。
- 从主 Agent 派发任务的路径短且清晰。

### Phase 4：视觉收敛与交互优化
目标：把面板打磨到产品可用状态。

交付内容：
- 任务卡、Agent 卡、状态徽标统一视觉语言。
- 明确 hover / focus / disabled 状态。
- 优化可读性与信息密度。

验收标准：
- 右侧面板不再“信息堆砌”。
- 用户无需查看文档即可理解常用操作。

## 8. 技术实施建议

### 8.1 前端拆分建议
- 将多 Agent 面板拆出子组件：
  - `multi-agent-mode-switch`
  - `multi-agent-lead-agent-card`
  - `multi-agent-task-board`
  - `multi-agent-team-list`
  - `multi-agent-event-timeline`
- 这样可以降低单文件复杂度，也更容易控制样式预算。

### 8.2 状态管理建议
- 任务数据与 agent 数据分别维护。
- 由 orchestrator 统一输出组合 VM。
- UI 只消费 VM，不直接拼装业务关系。

### 8.3 事件驱动建议
- 所有任务变更都通过事件总线记录。
- 事件时间线作为唯一回溯入口之一。
- 失败原因要尽量结构化，方便界面提示和排障。

## 9. 风险与注意事项

1. **概念迁移风险**
   - 当前页面已有很多功能，重构过程中容易把执行模式、任务、Agent 关系再次混淆。

2. **样式预算风险**
   - `workbench.page.scss` 已经接近预算上限，新增样式最好分组件拆离。

3. **兼容性风险**
   - tmux / WSL2 / iTerm2 行为差异很大。
   - 任务分配不应依赖某一种后端才能成立。

4. **事件噪声风险**
   - 事件太多会导致用户不知道先看哪条。
   - 需要分级：阻塞、失败、普通流转、调度。

## 10. 验收清单

重构完成后，至少满足以下条件：

- [ ] 用户能明确区分“执行模式”和“任务分配”。
- [ ] 主 Agent 在 UI 中显式可见。
- [ ] 任务卡可编辑，且可指定负责人/目标 Agent。
- [ ] Agent 卡能显示当前任务与状态。
- [ ] 事件时间线能回看任务分配与 Agent 流转。
- [ ] Windows 下 tmux 仍能通过 WSL2 正确执行。
- [ ] 页面样式不过度膨胀，构建预算可接受。

## 11. 推荐执行顺序

1. 先落数据模型与状态定义。
2. 再重构右侧面板 UI 结构。
3. 再接入任务分配与主 Agent 语义。
4. 最后做样式收口与交互打磨。

---

如果后续要开始实施，建议先从 **Phase 0 + Phase 1** 一起做，这样能最快把“执行模式”和“任务分配”的概念彻底拆开，避免 UI 再次混乱。
