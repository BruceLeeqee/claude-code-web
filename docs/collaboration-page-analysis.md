# 协作页面功能分析与优化建议

日期：2026-04-17
范围：`zyfront-desktop/src/app/features/prototype/collaboration/collaboration.page.*`

## 1. 页面当前职责

协作页面承担的是一个多 Agent 协作工作台，当前已经覆盖以下能力：

- 创建 Team
- 补齐执行者
- 分派任务
- 观察 Agent 回传
- 保存 / 恢复快照
- 查看事件流与反馈
- 切换运行模式
- 跳转到 workbench 定位单个 Agent

从产品定位上看，它不是一个简单列表页，而是“协作流程入口 + 编排控制台 + 恢复观测面板”的组合页。

## 2. 当前已有功能梳理

### 2.1 顶部输入与全局控制

当前顶部提供了：

- Team 名称输入
- 执行者数量选择
- 运行模式切换
- 创建 Team 按钮
- 刷新按钮

这部分相当于页面的“初始化层”，负责建立协作上下文。

### 2.2 流程引导区

页面中有一套四步流程：

1. 创建 Team
2. 补齐执行者
3. 分配任务
4. 观察回传

这部分已经开始向“向导式流程”靠拢，是当前页面最接近正确方向的主线。

### 2.3 Team 概览

页面展示了：

- 当前 Team 名称
- 模式
- 成员数量
- 后端状态
- 在线 Agent 数
- 执行中数量
- 待响应数量
- 任务总数

这部分是状态总览，但当前和流程主线之间的层级关系还不够清晰。

### 2.4 执行者与任务

当前页面同时保留了：

- 任务卡列表
- 任务分派
- Agent 角色映射
- 选中任务详情
- 任务状态变化

这说明页面已经具备“任务驱动协作”的基础，但呈现方式还比较分散。

### 2.5 恢复与观测

页面还有：

- 操作反馈
- 恢复快照保存 / 恢复
- 事件分级
- 最近事件列表
- 运行观测指标

这部分能力是很有价值的，但目前占据了较多视觉注意力，导致主流程被打断。

### 2.6 Agent 操作

页面还支持：

- 停止 Agent
- 推进 Agent
- 跳转到 workbench
- 查看终端相关信息

这些属于低频动作，适合后置到详情区或抽屉中，不适合和主流程并列展示太多。

## 3. 当前逻辑混乱的主要原因

### 3.1 主线不够唯一

页面同时存在多个“看起来都重要”的中心：

- 创建流程
- Team 概览
- 执行者面板
- 任务编排
- 恢复观测
- 事件流
- 技能面板
- 角色映射
- 总结区

这些模块都合理，但没有明确的主次顺序，用户会感觉页面“哪里都重要”，结果就是哪里都不聚焦。

### 3.2 结构更像信息堆叠，而不是流程推进

虽然现在已经有流程 step，但页面整体仍然更像“多个卡片集合”，不是一个真正的编排向导。

用户体验上会出现：

- 看到很多模块，但不知道先做什么
- 创建 Team 成功后，页面仍有太多并列内容干扰下一步
- 任务、Agent、事件、恢复都在同一视觉层级，缺少流转关系

### 3.3 创建 Team 与后续动作耦合不够强

目前成功后会推进到下一阶段，但页面上仍存在多种入口同时可用：

- 一键创建 Team
- 补齐执行者
- 新建任务
- 保存快照
- 继续当前阶段
- 恢复上次状态

这会让用户不知道此刻最推荐做哪一个。

### 3.4 任务和 Agent 的关系表达不够直观

当前任务列表、执行者面板、角色映射是分开展示的，虽然数据有关联，但视觉上没有形成一个统一的“任务流转路径”。

用户更容易看到“卡片”，而不是“谁在处理什么任务，任务如何推进”。

### 3.5 观测信息暴露过多

事件流、反馈、恢复快照、运行指标、操作结果都在页面中可见，信息密度太高，削弱了“编排中心”的可操作性。

## 4. 适合保留的核心能力

从用户价值看，以下能力建议继续保留：

- Team 创建
- 执行者补齐
- 任务分派
- 任务选中与详情查看
- 恢复快照
- 事件流
- Agent 跳转
- 运行模式切换

这些是实用能力，不应删除，只应重新分层。

## 5. 建议的页面主线

建议把页面重新定义成 4 层主线：

### 5.1 第 1 层：创建向导

只负责回答“现在该做什么”：

- 输入 Team 名称
- 选择目标执行者数
- 选择运行模式
- 点击创建 Team
- 成功后自动进入补齐阶段

### 5.2 第 2 层：当前状态摘要

只保留最少信息：

- Team 是否已创建
- 当前阶段
- 执行者是否充足
- 是否存在阻塞
- 下一步建议是什么

### 5.3 第 3 层：主工作区

这是页面最重要的区域，只展示一个主任务：

- 左侧：任务流 / 执行者流转
- 右侧：当前选中任务或节点详情

### 5.4 第 4 层：低频观测与恢复

包括：

- 事件流
- 操作反馈
- 恢复快照
- 指标
- 调试信息

建议折叠、收口或作为底部区域存在。

## 6. 页面优化建议

### 6.1 结构优化

建议把当前页面拆成 3 个明显区域，而不是很多并列卡片：

1. **创建流程区**
   - 像技能创建一样的向导
   - 只承担初始化与阶段切换

2. **编排工作区**
   - 任务卡
   - 执行者映射
   - 选中详情
   - 任务分派

3. **观测收口区**
   - 事件流
   - 反馈
   - 恢复快照
   - 运行指标

### 6.2 交互优化

- 创建 Team 成功后自动高亮“补齐执行者”
- 补齐完成后再显露任务分派入口
- 任务分派后自动聚焦选中的任务详情
- 出现阻塞时优先展示恢复动作，而不是只提示错误
- 推荐动作只保留一个主按钮，不要同时给太多相同级别的 CTA

### 6.3 信息层级优化

建议明确三类优先级：

- **一级**：当前阶段、主操作、下一步建议
- **二级**：任务、执行者、当前选中详情
- **三级**：事件、恢复、指标、调试信息

### 6.4 数据联动优化

建议让以下关系更强：

- Team 创建成功 → 自动推进补齐执行者
- 执行者足够 → 自动建议新建任务
- 任务被选中 → 自动聚焦任务详情
- 任务分派成功 → 自动切到观察状态
- 出现阻塞 → 自动突出恢复入口

### 6.5 视觉优化

- 主流程只保留一条明显视觉主线
- 低频模块使用更轻的卡片或折叠面板
- 减少同级卡片数量
- 把“状态”和“操作”分开，不要混排太多按钮

## 7. 推荐的重构顺序

如果继续重构，建议按下面顺序做：

1. **先删减无关区块**
   - 技能面板
   - 重复性的总结卡
   - 过多的并列状态卡

2. **再收敛成 3 大区**
   - 创建流程
   - 工作区
   - 观测区

3. **最后强化联动**
   - 创建后自动切阶段
   - 任务选中联动右侧
   - 阻塞时突出恢复

## 8. 建议增加 Tab 分层展示

考虑到页面能力较多，建议把“全部展示”改成“按节点分 Tab 展示”。这样页面不会同时暴露所有卡片，用户只会看到当前节点需要的内容。

同时，Tab 的视觉样式建议直接参考“新建技能”流程页：左侧流程导航 / 中央内容区 / 右侧详情区的布局关系、深色卡片语义、主按钮高亮、选中态边框与渐变底色，都保持一致。协作页只替换成“创建 Team → 补齐执行者 → 分派任务 → 观察回传”的业务内容。

### 8.1 Tab 分层建议

建议至少分为 4 个 Tab，并且只在对应节点显示对应内容：

1. **创建**
   - Team 名称
   - 执行者数量
   - 运行模式
   - 创建 Team
   - 创建流程向导

2. **补齐**
   - Team 状态
   - 执行者面板
   - 角色映射
   - 补齐执行者动作

3. **分派**
   - 任务分配流
   - 任务详情
   - 任务分派按钮
   - 任务相关反馈

4. **观察**
   - 恢复快照
   - 操作反馈
   - 事件流
   - 运行观测

### 8.2 Tab 显示原则

- 当前节点只显示当前 Tab
- 已完成节点可回看，但默认不展开
- 阻塞节点强制展示恢复相关 Tab
- 观测类信息默认只在“观察”Tab 中集中出现
- 样式尽量与“新建技能”流程页统一：同样的步骤导航、卡片边界、选中态和按钮层级

### 8.3 Tab 带来的好处

- 减少首屏噪声
- 降低用户认知负担
- 让页面更接近技能创建的分步体验
- 把“状态切换”显性化
- 避免所有卡片同时抢焦点

## 9. 推荐的详细页面信息架构

为了让协作页真正成为“编排中心”，建议把它的信息架构重新定义为下面四层。

### 9.1 顶层：任务目标与阶段

这一层只回答两个问题：

- 现在处于哪个阶段
- 下一步最应该做什么

建议固定展示：

- 当前 Team 名称
- 当前运行模式
- 当前阶段标签
- 当前阻塞原因（如果有）
- 下一步推荐动作

这一层的目标是让用户不用阅读下面的卡片，就能立即知道“该干什么”。

### 9.2 第二层：协作主路径

这是页面真正的主工作区，只展示与协作主线强相关的信息：

- 任务流转
- 执行者列表
- 角色分配
- 当前选中任务
- 任务状态变化

这里建议采用“左流转、右详情”的稳定布局：

- 左侧显示任务和 Agent 的关系
- 中间显示当前阶段中的流程节点
- 右侧显示当前选中对象详情

这样可以把“谁在处理什么”表达得更直接。

### 9.3 第三层：辅助观测区

这层包含低频但重要的调试与回溯能力：

- 事件流
- 操作反馈
- 运行指标
- 恢复快照
- 最近错误 / 警告

建议默认折叠，或者放在底部标签区，不要和主流程同屏争夺注意力。

### 9.4 第四层：危险与高成本操作区

这里放那些不该被误触的操作：

- 停止 Agent
- 推进 Agent
- 恢复快照
- 清空 / 重置类操作
- 跳转到 workbench

这些操作建议：

- 使用次级按钮样式
- 放在详情抽屉里
- 需要二次确认
- 仅在选中具体对象时出现

## 10. 更完整的交互设计建议

### 10.1 首次进入页面

首次进入时，页面应该强引导用户完成初始化，而不是直接把所有信息铺开。

建议默认流程：

1. 显示创建 Team 的主卡片
2. 提示当前还没有协作上下文
3. 用户输入 Team 名称和目标人数
4. 点击创建后自动进入“补齐执行者”阶段

如果页面已有 Team 状态，则应直接恢复到上次阶段，不再要求用户重复创建。

### 10.2 创建成功后的行为

创建成功后建议自动发生三件事：

- 高亮下一步入口
- 滚动到执行者补齐区域
- 在顶部摘要中更新阶段状态

这样用户不会在“创建完成后不知道接下来点哪里”。

### 10.3 执行者补齐后的行为

执行者补齐成功后，页面应当：

- 展示任务分派区
- 提示当前空闲 Agent 数量
- 给出推荐任务分配方案
- 自动聚焦到第一个待处理任务

### 10.4 任务分派后的行为

任务分派一旦成功，页面建议：

- 自动切换到观察态
- 展示最新回传摘要
- 把刚分派出去的任务卡标记为高亮
- 把相关事件推到事件流顶部

### 10.5 阻塞态的行为

当系统检测到阻塞时，不要只展示红色错误文本，而应该直接给出“修复路径”：

- 是否缺少执行者
- 是否需要恢复快照
- 是否 Agent 已停止
- 是否需要重新分配任务

也就是说，错误提示要附带“下一步怎么做”。

## 11. 状态机视角的页面重构建议

如果把协作页当作一个状态机，它其实只有少数几个核心状态：

1. **未创建**
2. **创建中**
3. **补齐执行者**
4. **等待分派**
5. **执行中**
6. **观察中**
7. **阻塞中**
8. **可恢复**

当前页面的问题，是把这些状态同时暴露出来了。

更好的方式是：

- 每次只突出一个主状态
- 次要状态收进摘要条
- 只有进入相关状态时才展示对应模块

例如：

- 处于“未创建”时，只展示创建区
- 处于“补齐执行者”时，只展示执行者区
- 处于“执行中”时，只展示任务和观察区
- 处于“阻塞中”时，只展示恢复与修复建议

## 12. 推荐的数据联动规则

建议把页面联动进一步明确成下面几条规则。

### 12.1 Team 创建规则

- 创建成功后自动写入阶段状态
- 自动清理旧任务选择
- 自动重置当前详情视图

### 12.2 Agent 补齐规则

- 数量不足时显示缺口
- 补齐后自动解锁任务分派区
- 若存在角色冲突，直接标红冲突 Agent

### 12.3 任务分派规则

- 任务被分派后自动附着到对应 Agent
- 当前任务卡状态立即更新
- 相关事件立即进入事件流

### 12.4 观测规则

- 只有发生变更时才提升事件优先级
- 重复事件自动合并
- 非关键日志默认折叠

### 12.5 恢复规则

- 有快照时优先恢复最近一次有效状态
- 恢复前提示将覆盖当前未保存状态
- 恢复后自动回到对应阶段

---

## 13. 可以直接落地的 UI 改动清单

如果要快速改进页面体验，建议优先做下面这些动作：

1. **缩减首屏卡片数量**
   - 只保留创建、状态摘要、主工作区

2. **把次级信息收进折叠区**
   - 事件流
   - 反馈
   - 恢复快照
   - 调试信息

3. **统一主按钮风格**
   - 每个阶段只保留一个明显的主按钮

4. **强化当前阶段高亮**
   - 当前步骤使用更强的背景、边框或标签

5. **把危险操作拆出去**
   - 避免和普通操作放在一起

6. **增加空状态引导**
   - 没有 Team 时给出明确说明
   - 没有任务时给出创建任务入口
   - 没有事件时显示“等待回传”

## 14. 进一步建议：按节点拆分为可复用子模块

如果后续要继续重构页面，最好把页面拆成以下可复用模块：

- `collaboration-create-panel`
- `collaboration-stage-summary`
- `collaboration-agent-board`
- `collaboration-task-flow`
- `collaboration-observation-panel`
- `collaboration-recovery-panel`
- `collaboration-danger-actions`

这样做的好处是：

- 逻辑更清晰
- 组件职责更单一
- 每个模块更容易维护
- 后续可以按 Tab 或按阶段动态装载

---

## 15. 结论

当前协作页面已经具备完整能力，但问题不在功能缺失，而在**信息组织方式过于分散**。

最关键的优化方向是：

- 把页面从“功能集合页”改成“流程向导页”
- 把“创建 Team”作为唯一主入口
- 把“补齐执行者 / 分派任务 / 观察回传”作为连续步骤
- 把恢复和观测降级为低频辅助信息
- **进一步按节点拆成 Tab，只展示当前节点需要的内容**
- **让每个阶段都有唯一主按钮、唯一主目标、唯一推荐动作**

这样页面逻辑会明显更顺，用户也更容易理解下一步该做什么。
