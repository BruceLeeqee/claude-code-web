# Zyfront Desktop Workbench 终端页面重构设计

> 目标：围绕 `zyfront-desktop` 的 workbench 终端主界面，单独梳理一份可落地的页面重构设计文档。
>
> 本文重点描述：左侧最近会话、中央终端主区、下侧默认输入框、右侧 AI 任务分栏，以及它们之间的事件联动、状态模型和分阶段落地计划。

---

## 1. 设计背景

当前 workbench 的终端页面已经承载了多种能力，包括会话、输出、任务、附件和协作状态，但在布局与交互上仍存在职责不够清晰、信息密度不均衡、状态同步分散等问题。

这份设计的目标不是简单“美化界面”，而是把终端页面升级为一个清晰的多智能体工作台：

- 用户能快速切换最近会话
- 用户能直接在底部输入框发起任务或附加文件
- 系统能在右侧清晰展示 AI 任务分工和进度
- 中央区域专注于输出、预览与执行结果
- 所有区域通过统一事件驱动联动

---

## 2. 页面布局目标

### 2.1 总体布局

建议采用三栏 + 底部输入区的结构：

- **左侧栏**：最近会话 / 会话管理
- **中央主区**：终端输出与执行流
- **右侧栏**：AI 任务分栏 / 角色状态 / 计划视图
- **底部固定区**：默认输入框，支持文本、文件、图片、多模态附加内容

### 2.2 设计原则

1. **会话优先**：左侧先解决“我在哪个会话里工作”。
2. **输入优先**：底部输入框常驻，降低用户发起成本。
3. **任务优先可见**：右侧始终展示 AI 正在做什么。
4. **输出专注**：中央区域只承担执行过程与结果展示。
5. **联动一致**：任何状态变化都通过统一事件总线驱动。

---

## 3. 左侧最近会话设计

### 3.1 设计目标

让用户快速定位和恢复最近工作上下文，不必在多个页面之间跳转。

### 3.2 核心内容

左侧栏建议包含：

- 最近会话列表
- 会话搜索 / 筛选
- 会话固定 / 归档
- 会话状态标识
- 未读任务或告警提示

### 3.3 会话卡片信息

每个会话卡片建议展示：

- 会话名称
- 最近更新时间
- 当前运行状态
- 关联任务数量
- 关键标签或项目名

### 3.4 交互行为

- 点击会话卡片切换当前上下文
- 悬停可展示摘要和最近输出
- 支持固定到顶部
- 支持快速归档和删除
- 支持按项目、状态、时间筛选

### 3.5 状态要求

左侧会话列表必须订阅统一的 `session` 投影结果，不能直接推断运行中的 agent 状态。

---

## 4. 底部默认输入框设计

### 4.1 设计目标

把输入动作做成最常用、最稳定、最少步骤的入口。

### 4.2 输入框能力

输入框应支持：

- 文本输入
- 快捷命令
- 文件上传
- 图片附件
- 草稿缓存
- 回车发送 / 组合键发送
- 上下文提示与快捷建议

### 4.3 输入框结构

建议包含以下区域：

- 主输入区
- 附件 chips 区
- 快捷操作区
- 发送按钮
- 状态提示区

### 4.4 交互细节

- 默认常驻于底部
- 支持多行自动扩展，但不遮挡中央输出
- 附件上传后以 chip 形式展示，可删除、可预览
- 支持输入草稿恢复
- 支持快捷命令触发任务模板

### 4.5 输入协议

输入内容建议统一封装为 `InputEvent`，至少包含：

- `sessionId`
- `text`
- `attachments`
- `command`
- `timestamp`
- `source`

---

## 5. 中央终端主区设计

### 5.1 设计目标

中央区域承载任务执行过程、终端日志、结果展示和错误定位，是工作台最核心的“过程视图”。

### 5.2 主要内容

中央主区建议支持：

- 终端输出流
- 执行日志
- 结果卡片
- diff 预览
- 文件预览
- 图片预览
- 错误提示与定位

### 5.3 视觉策略

- 终端流保持信息密度，但要有层次
- 结果类内容用卡片化样式
- 输出流与结构化卡片共存，但时间轴一致
- 长输出采用虚拟滚动和折叠策略

### 5.4 交互行为

- 点击错误项跳转到对应任务或日志段落
- 支持复制输出
- 支持展开 / 折叠长输出
- 支持滚动定位到最新消息
- 支持在输出区直接打开预览内容

---

## 6. 右侧 AI 任务分栏设计

### 6.1 设计目标

让用户始终知道系统当前正在处理哪些任务、由哪些角色承担、进展如何、是否存在风险。

### 6.2 任务分栏内容

右侧栏建议包含：

- 当前任务概览
- 任务树 / DAG 视图
- 角色分工卡片
- 任务进度条
- 风险与阻塞提示
- 任务操作入口

### 6.3 角色卡片建议

角色卡片可展示：

- role 名称（leader / planner / executor / reviewer / researcher 等）
- 绑定模型
- 当前状态
- 最近心跳
- 负责任务数
- 是否阻塞

### 6.4 交互行为

- 点击任务节点定位中央输出或日志
- 支持展开任务依赖关系
- 支持查看 agent 负责范围
- 支持对阻塞任务进行干预
- 支持手动触发重新规划

---

## 7. 联动与状态同步

### 7.1 设计原则

四个区域必须通过统一事件总线联动：

- 左侧会话切换
- 底部输入提交
- 中央输出刷新
- 右侧任务状态更新

### 7.2 推荐事件

建议至少存在以下事件：

- `session.created`
- `session.resumed`
- `session.closed`
- `input.submitted`
- `task.planned`
- `task.started`
- `task.blocked`
- `task.completed`
- `agent.created`
- `agent.started`
- `agent.idle`
- `agent.failed`
- `model.routed`
- `memory.synced`

### 7.3 状态投影

UI 只订阅事件投影结果，不直接读内部执行对象。这样可以避免：

- 页面状态漂移
- 多处重复计算
- 运行态与显示态不一致

---

## 8. 响应式与布局适配

### 8.1 宽屏

- 左侧会话栏固定显示
- 右侧任务栏固定显示
- 中央区域宽度自适应

### 8.2 中屏

- 左右栏可折叠为抽屉或悬浮面板
- 中央区域优先保留

### 8.3 窄屏

- 默认只显示中央区域与底部输入框
- 左右栏通过抽屉打开
- 任务状态和会话列表进入二级入口

---

## 9. 关键状态模型

建议统一以下核心状态：

- `sessionState`
- `inputState`
- `taskState`
- `agentState`
- `outputState`
- `attachmentState`
- `layoutState`

这些状态需要明确来源、生命周期和同步方式。

---

## 10. 分阶段实施计划

### Phase 0：终端工作台基线冻结

**目标**：先把 workbench 终端页面的布局、状态模型和交互边界冻结，避免后续重构时 UI 与运行时同时变化。

#### 关键工作
- 确认左侧最近会话、中央终端、底部输入框、右侧 AI 任务栏的职责边界
- 统一 session / task / attachment / input event 的数据结构
- 定义终端输出卡片、任务卡片、附件卡片的投影协议
- 明确宽屏、中屏、窄屏的布局切换规则

#### 交付物
- 终端工作台布局草图
- 组件职责说明
- 输入事件协议草案
- 会话/任务状态机草案

#### 验收标准
- 设计层面不存在职责重叠
- 任何页面状态都能映射到统一领域模型

### Phase 1：左侧最近会话 + 底部输入框 MVP

**目标**：先让用户可以快速切换会话，并用统一输入框完成文本/文件/图片输入。

#### 关键工作
- 实现最近会话列表与筛选
- 实现会话恢复与固定/归档操作
- 实现底部输入框的多模态附件能力
- 实现草稿缓存与快捷命令入口

#### 交付物
- 最近会话组件
- 输入框组件
- 附件 chip 组件
- 草稿恢复机制

#### 验收标准
- 用户可在不离开 workbench 的情况下完成会话切换和附件输入

### Phase 2：右侧 AI 任务分栏 MVP

**目标**：把任务计划、执行、审查、失败恢复统一呈现在右侧栏。

#### 关键工作
- 任务概览头部
- 任务树 / DAG 视图
- 角色状态卡片
- 进度条与风险提示
- 任务操作区

#### 交付物
- AI 任务分栏组件
- 任务状态联动逻辑
- 节点点击定位能力

#### 验收标准
- 用户能通过右侧栏理解任务进度、角色分工与阻塞原因

### Phase 3：中央终端主区内部效果增强

**目标**：让中央输出区兼容终端流、智能建议卡、预览卡和错误定位。

#### 关键工作
- 输出流统一时间轴
- 建议卡片与终端日志共存
- diff / 图片 / 文件预览
- 长输出虚拟滚动
- 错误定位与复制操作

#### 交付物
- 统一输出投影层
- 卡片渲染器
- 预览组件

#### 验收标准
- 中央区域可承载复杂任务的完整执行轨迹

### Phase 4：事件驱动联动与状态同步

**目标**：打通左侧会话、底部输入、中央输出、右侧任务栏之间的事件同步。

#### 关键工作
- 统一事件总线接入
- 会话状态同步
- 任务状态同步
- 附件状态同步
- 错误与重试链路

#### 交付物
- 状态投影层
- 事件路由与订阅机制
- 统一 loading / error / success 反馈

#### 验收标准
- 页面内任何关键变化都可以在 200ms 级别内联动更新

### Phase 5：自治协作增强

**目标**：引入自动规划、自动扩容、自动恢复，让 workbench 具备更强的多智能体协作能力。

#### 关键工作
- 任务自动拆解
- 自动创建 agent
- 自动接管失败任务
- 自动回收无效 agent
- 模型路由联动

#### 交付物
- 自治协作策略
- 自动扩容规则
- 失败恢复面板

#### 验收标准
- 复杂任务可以在较少人工干预下完成闭环

---

## 11. 风险与对策

### 11.1 风险：布局职责混乱

**对策**：先冻结四大区域的职责边界，再做 UI 细化。

### 11.2 风险：输入框能力扩张过快

**对策**：先做文本 + 附件的稳定入口，再逐步加入更多输入类型。

### 11.3 风险：事件同步引入复杂度

**对策**：统一事件协议与状态投影，UI 不直接读运行时对象。

### 11.4 风险：右侧任务栏信息过载

**对策**：默认只显示关键状态，详细内容通过展开查看。

### 11.5 风险：多会话切换导致上下文串线

**对策**：会话级隔离、输入级绑定、输出级归属必须严格一致。

---

## 12. 结论

这份 workbench 终端页面重构设计的核心，不是把页面做得更“花”，而是把信息与职责重新分配清楚：

- 左侧负责会话
- 底部负责输入
- 中央负责执行输出
- 右侧负责任务与协作状态

当这四个区域都被统一到事件驱动和领域模型之后，workbench 才能真正成为一个可扩展、可恢复、可自治的多智能体终端工作台。
