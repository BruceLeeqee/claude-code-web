# 多 Agent 编排中心动态工作流重构计划

> 目标：将当前多 Agent 协作流程从“纵向堆叠、依赖上下滚动查看”的信息展示方式，升级为“类似截图中的工作流式动态流转”体验。
>
> 核心要求：
> - 流程以步骤驱动、状态驱动、事件驱动的方式推进
> - 用户不需要上下滚动页面就能理解当前协作进度
> - 重点优化流转感、可读性、操作效率与异常恢复体验
> - 同步优化页面结构、交互、样式、空态、反馈和可观测性

---

## 1. 背景与问题定义

### 1.1 当前体验问题

现有多 Agent 协作中心更接近“内容面板 + 事件列表”的组合，而不是“任务流工作台”。主要问题包括：

1. **流程是静态的**
   - Agent、任务、事件、输出分散展示
   - 用户需要自行拼接上下文，理解谁在做什么

2. **依赖上下滚动阅读**
   - 关键流程信息分布在较长页面中
   - 用户需要不断滚动查看不同模块，认知负担高

3. **协作链路不够直观**
   - 不容易看出“创建任务 → 分配 → 执行 → 回传 → 汇总”的主线
   - 事件虽多，但缺少一个“当前所处步骤”的统一视角

4. **样式层级不足**
   - 主流程、辅助信息、日志信息的视觉层次不够明确
   - 缺少“正在流转”的动效、连接关系和状态反馈

5. **异常恢复不成体系**
   - 异常后用户仍然需要自己寻找入口处理
   - 没有把恢复动作嵌入流程节点中

### 1.2 重构目标

把多 Agent 编排中心改造成一个“可动态流转的工作流编排界面”，让用户始终看到：

- 当前处于哪一步
- 哪个 Agent 正在处理
- 下一步会发生什么
- 出错后应该点哪里恢复

---

## 2. 设计原则

### 2.1 流程优先，而不是信息优先

页面第一展示层必须是“流程状态”，不是“数据列表”。

### 2.2 主线前置，细节后置

默认只展示最关键的协作主线，详细事件、日志、历史记录通过展开、侧边抽屉或二级视图查看。

### 2.3 动态流转可感知

任何状态变化都应该有可见反馈，包括：
- 进度变化
- 节点高亮
- 连线流动
- 任务卡迁移
- 状态标签更新

### 2.4 无需滚动即可掌控局面

在默认视口内完成以下事情：
- 看懂当前流程
- 看到 Agent 状态
- 发起主要操作
- 观察最近事件
- 进入恢复动作

### 2.5 异常即节点，不是弹窗噪声

异常要落回到流程节点上，成为可处理的状态，而不是仅仅弹出错误提示。

---

## 3. 目标体验形态

### 3.1 视觉形态

参考截图所表达的体验方向，建议采用：

- **顶部步骤条 / 流程阶段条**：展示当前协作流程所处阶段
- **中部动态工作流画布**：展示任务在多个 Agent 之间的流转
- **左右辅助面板或底栏**：展示状态、操作、说明、最近事件
- **底部固定操作区**：只保留最关键的 CTA

### 3.2 交互形态

用户在页面中应形成如下感受：

- 流程像“被推进”一样在走
- Agent 像“节点”一样接力处理
- 任务像“卡片”一样在节点间移动
- 状态变化像“轨迹”一样清晰

---

## 4. 核心信息架构重构

### 4.1 页面分层

建议将编排中心拆成 4 个稳定层级：

#### A. 流程阶段层
用于回答“现在处于哪一步”。

展示内容：
- 当前阶段名称
- 阶段进度
- 阶段状态（待处理 / 进行中 / 已完成 / 阻塞）
- 下一步建议

#### B. 工作流流转层
用于回答“任务正在如何流转”。

展示内容：
- 任务节点
- Agent 节点
- 连线关系
- 当前执行点
- 完成态与阻塞态

#### C. 任务与 Agent 详情层
用于回答“谁在做、做了什么、做到哪了”。

展示内容：
- 当前负责人
- Agent 角色与状态
- 最近动作
- 当前子任务
- 结果摘要

#### D. 观测与恢复层
用于回答“出了问题怎么办”。

展示内容：
- 最近事件
- 错误原因
- 重试入口
- 重新分配入口
- 回滚 / 恢复入口

---

## 5. 动态工作流模型

### 5.1 流程主线建议

建议把多 Agent 协作抽象为以下阶段：

1. **准备阶段**
   - 创建团队 / 初始化环境
   - 检查可用 Agent
   - 选择运行模式

2. **编排阶段**
   - 生成协作任务
   - 拆分子任务
   - 指派执行者

3. **执行阶段**
   - Agent 开始处理
   - 任务卡进入流转中状态
   - 事件流记录过程

4. **汇总阶段**
   - 各 Agent 回传结果
   - Leader 汇总输出
   - 形成最终结论

5. **收敛阶段**
   - 标记完成
   - 保存快照
   - 清理临时状态

### 5.2 节点类型

建议统一定义 4 类节点：

- **控制节点**：Team / Leader / 全局状态
- **执行节点**：Teammate / 执行中的 Agent
- **任务节点**：任务卡、子任务卡、检查点
- **恢复节点**：失败、重试、阻塞、修复

### 5.3 节点状态

每个节点统一支持以下状态：

- 待命
- 进行中
- 已完成
- 阻塞
- 异常
- 重试中

### 5.4 流转关系

任务流转必须能清晰表达：

- 谁发起
- 谁执行
- 谁确认
- 谁汇总
- 谁负责恢复

## 6. 页面结构建议

### 6.1 推荐结构：固定高度 + 分区布局

为避免上下滚动，建议整体采用固定视口内布局：

- **顶部**：页面标题、全局状态、阶段条
- **中间左侧**：工作流画布 / 主流程节点
- **中间右侧**：当前节点详情 / Agent 状态 / 操作区
- **底部**：最近事件横条 / 轻量日志 / 快捷操作

### 6.2 推荐结构示意

- 顶部导航区
- 流程阶段区
- 中央工作流主视图
- 右侧上下文详情区
- 底部动作栏

### 6.3 布局约束

- 默认内容必须完整落在一个屏幕高度内
- 需要展开的内容优先放在抽屉、浮层、侧栏，而不是继续纵向堆叠
- 长文本采用折叠、悬停提示、详情抽屉查看

---

## 7. 交互设计方案

### 7.1 流程阶段切换

阶段条不是纯展示，而是状态同步器：

- 点击阶段可查看该阶段内容
- 当前阶段高亮
- 已完成阶段可回看
- 阻塞阶段显示红色标记与恢复入口

### 7.2 工作流节点流转

任务卡或流程卡在不同节点间移动时，需具备：

- 进入动画
- 状态颜色变化
- 负责人标签切换
- 执行时间统计
- 结果摘要更新

### 7.3 任务卡交互

点击任务卡可：

- 查看详情
- 查看最近事件
- 重新分配
- 重试
- 标记完成
- 进入对应 Agent

### 7.4 Agent 节点交互

点击 Agent 节点可：

- 查看当前状态
- 查看最近任务
- 查看会话绑定
- 进入终端/执行上下文
- 执行停止/重启/重连

### 7.5 恢复动作交互

当流程阻塞时，在节点上直接显示恢复按钮：

- 重试
- 重新分配
- 跳过
- 修复会话
- 查看原因

## 8. 功能优化范围

### 8.1 多 Agent 协作主流程优化

#### 要做的事
- 把分散的协作动作收束成标准工作流
- 统一任务状态机
- 统一步骤推进逻辑
- 让 Leader 的决策、Teammate 的执行、Team 的统筹在视觉上形成闭环

#### 交付效果
- 用户能一眼看出主流程
- 不用翻找不同区域拼接上下文
- 每次状态变化都可追踪

### 8.2 任务调度优化

#### 要做的事
- 将“任务创建 / 分配 / 执行 / 回传 / 汇总”串成一个流程
- 明确每个阶段的责任 Agent
- 支持任务在节点间动态迁移

#### 交付效果
- 任务不是静态列表，而是流转中的对象
- 任务阶段更清晰，执行过程更直观

### 8.3 Agent 卡片优化

#### 要做的事
- 简化卡片信息密度
- 强化状态、角色、当前任务、最近事件
- 用更强的视觉层次区分 Leader 和 Teammate

#### 交付效果
- 看卡片就知道是谁、在干什么、是否异常

### 8.4 事件与日志优化

#### 要做的事
- 默认只保留关键业务事件
- 技术日志折叠到二级视图
- 事件与工作流节点建立关联

#### 交付效果
- 用户不用在一堆日志里找线索
- 事件成为流程辅助，而不是主视觉负担

## 9. 样式优化方案

### 9.1 视觉风格方向

建议采用：

- 深色主题为主
- 高对比但低噪声的层级区分
- 柔和霓虹感状态色
- 节点与连线有轻量发光或渐变感
- 当前流转位置有更强的视觉焦点

### 9.2 关键视觉元素

#### 流程阶段条
- 圆点 / 编号 / 阶段名称
- 当前阶段高亮
- 已完成阶段弱高亮
- 阻塞阶段红色警示

#### 工作流连线
- 连线采用动态流光或进度渐变
- 已流转部分与未流转部分区分明显
- 连接方向清晰

#### 任务节点卡片
- 圆角卡片
- 状态标签
- 负责人头像/图标
- 结果摘要
- 进度条

#### 详情面板
- 采用分组标题
- 折叠次要内容
- 保持与主画布一致的视觉语言

### 9.3 组件样式规范

- 统一间距节奏
- 统一状态色板
- 统一按钮层级
- 统一卡片阴影与边框
- 统一标题字重与层级

### 9.4 空态与异常态样式

空态要能回答：
- 现在为什么空
- 我该做什么

异常态要能回答：
- 哪个节点异常
- 为什么异常
- 我怎么修

## 10. 关键业务状态设计

### 10.1 流程状态

建议统一为：
- 未开始
- 编排中
- 执行中
- 汇总中
- 已完成
- 阻塞
- 异常

### 10.2 任务状态

建议统一为：
- 待分配
- 已分配
- 执行中
- 等待确认
- 已完成
- 失败
- 重试中

### 10.3 Agent 状态

建议统一为：
- 空闲
- 工作中
- 等待输入
- 连接中
- 后台保活
- 异常

### 10.4 恢复状态

建议统一为：
- 可恢复
- 正在恢复
- 恢复成功
- 恢复失败

---

## 11. 数据与状态流设计

### 11.1 数据驱动原则

页面必须由统一状态源驱动，避免：

- 一个组件一套状态
- 任务、Agent、事件彼此不同步

### 11.2 推荐状态结构

建议将状态拆成以下几类：

- `workflowStage`
- `workflowNodes`
- `workflowEdges`
- `selectedNode`
- `selectedAgent`
- `taskQueue`
- `recentEvents`
- `recoveryActions`

### 11.3 同步原则

任何状态变化都应同步更新：

- 节点状态
- 卡片状态
- 右侧详情
- 底部事件
- 操作按钮可用性

---

## 12. 交付路径与实施顺序

### Phase 1：布局重构

#### 目标
先把页面从“纵向堆叠”变成“固定视口内分区布局”。

#### 任务
- 规划新的区域划分
- 固定顶部、主视图、右侧详情、底部动作栏
- 清理冗余纵向内容

#### 验收
- 默认视口内不需要滚动就能完成核心操作
- 页面主线清楚

### Phase 2：工作流视图重建

#### 目标
把多 Agent 协作主流程改造成节点流转视图。

#### 任务
- 定义节点类型与连接关系
- 实现当前节点高亮
- 实现状态迁移视觉反馈

#### 验收
- 用户能直接看懂协作流转
- 任务执行过程可视化

### Phase 3：操作与恢复整合

#### 目标
把常用操作嵌入节点和卡片中。

#### 任务
- 任务分配、重试、重连、重新分配等动作接入节点
- 异常节点提供恢复路径

#### 验收
- 操作不需要在多个区域中找入口
- 阻塞状态下也能立即找到处理按钮

### Phase 4：样式统一与动效增强

#### 目标
让工作流“活起来”。

#### 任务
- 节点动效
- 连线动效
- 状态切换动画
- 卡片层级视觉优化

#### 验收
- 页面有明显“流转感”
- 视觉统一且不杂乱

### Phase 5：观测与可恢复优化

#### 目标
补齐异常、历史、回放、恢复能力。

#### 任务
- 最近事件与工作流节点绑定
- 支持恢复动作
- 支持历史阶段回看

#### 验收
- 出错后可操作
- 用户不会被动等待或猜测

---

## 13. 验收标准（DoD）

本次重构完成后，至少满足以下标准：

1. 默认页面视口内可理解当前协作流程
2. 用户不需要上下滚动就能完成主要操作
3. 任务、Agent、阶段、事件之间有明确关联
4. 工作流具备明显的动态流转感
5. 异常节点可直接恢复，不依赖额外排查
6. 样式层级清晰，主次信息分明
7. 新用户第一次接触页面时能快速看懂“现在到哪一步了”

---

## 14. 风险与注意事项

### 14.1 风险

- 过度强调动态效果，导致页面花哨或分散注意力
- 节点状态过多，反而难以理解
- 详情层级设计不当，仍然会回到纵向滚动问题
- 动态流转与数据状态不同步，造成视觉假象

### 14.2 注意事项

- 动效必须服务于理解，不是为了炫技
- 默认态应该足够简洁
- 细节展示必须可展开、可收起、可聚焦
- 保持统一状态机，避免节点状态碎片化

---

## 15. 建议的下一步开发顺序

1. 先确定新的页面布局框架
2. 再定义工作流阶段和节点模型
3. 再重做任务/Agent 流转展示
4. 再将操作入口嵌入节点
5. 最后统一样式、动效和异常恢复

---

## 16. 结论

这次改造的本质不是“把页面做得更花”，而是把多 Agent 协作从“静态信息堆叠”升级为“动态工作流编排”。

最终要让用户获得这样的体验：

- 流程在动
- 节点在接力
- 任务在流转
- 状态可追踪
- 异常可恢复
- 全程无需依赖上下滚动去理解当前局面
