# Workbench 多智能体体验优化建议计划（产品版）

> 角色视角：产品经理  
> 适用范围：当前 `docs/workbench-multi-agent-operation-manual.md` 覆盖能力  
> 参考基线：ClaudeCode 多智能体语义（leader + teammate）、以及成熟 Agent 产品的“低摩擦协作 + 高可观测 + 可恢复”体验

---

## 1. 执行摘要（TL;DR）

当前实现已经具备多智能体基础闭环（spawn/stop/kill、事件时间线、Tab=Agent、会话持久化），但仍存在典型“工程可用、产品未成熟”的差距：

1. **信息负载过高**：Workbench 右栏 + Agent Tab 的状态信息分散，用户难以快速判断“现在该做什么”。
2. **反馈层级不足**：关键动作（Spawn/Stop/Kill/Backend blocking）缺少统一的成功/失败反馈与恢复建议。
3. **任务导向不够**：更像“控制台”，还不是“协作工作流”。
4. **稳定性心智未建立**：缺少可预期的恢复、重试、排障路径。

建议按 4 个里程碑推进，8~10 周达到“团队可持续使用”的产品体验水位。

---

## 2. 产品目标与北极星指标

## 2.1 北极星目标

把 Workbench 从“多智能体操作台”升级为“多智能体协作工作台”。

## 2.2 关键指标（建议）

- **任务完成效率**
  - 从创建第一个 agent 到完成一次协作任务的中位时长（TTV）
- **操作成功率**
  - Spawn/Stop/Kill 成功率
  - backend blocking 后用户恢复成功率
- **协作质量**
  - 单任务平均有效 agent 消息数（去噪后）
  - 用户手动干预次数（越少越好）
- **可恢复性**
  - 页面重开后会话恢复成功率
  - 异常后 3 分钟内恢复率

---

## 3. 现状评估（基于当前手册功能）

## 3.1 已有优势

- 已支持 `Tab=Agent`，符合 ClaudeCode 的独立上下文语义。
- 事件总线统一，Workbench 与 Collaboration 时间线同源。
- blocking + setup hints 已形成较好的“可解释失败”基础。
- 已有会话持久化，具备恢复能力雏形。

## 3.2 主要问题

- **入口与状态分散**：右栏、Tab、事件区都能操作，但缺少“主操作路径”。
- **缺少任务抽象层**：目前围绕 agent 操作，不围绕“目标任务阶段”。
- **反馈不统一**：部分失败静默，用户不知“是否成功/下一步是什么”。
- **信息噪声高**：事件行是技术事件，不是用户可读事件。
- **治理能力薄弱**：权限、速率、资源配额、异常恢复策略尚未产品化。

---

## 4. 对标体验原则（ClaudeCode/成熟 Agent 产品）

## 4.1 原则一：默认路径清晰

- 用户应在 3 次点击内完成：创建协作 → 指派任务 → 观察结果。

## 4.2 原则二：每一步都有反馈与建议

- 成功要确认，失败要解释，异常要给下一步。

## 4.3 原则三：多智能体“可控而不繁琐”

- 高级能力存在，但默认界面不暴露过多低频技术参数。

## 4.4 原则四：可恢复优先于花哨

- 会话、状态、任务应可恢复，可重放，可追踪。

---

## 5. 优化路线图与里程碑

## M1（第 1-2 周）：交互收敛与反馈统一

目标：让新用户“看得懂、点得对、知道发生了什么”。

范围：
1. **统一反馈机制**
   - 所有关键动作（Spawn/Stop/Kill/Send）引入统一 toast + 状态条反馈。
   - 失败信息结构化：原因、影响、建议动作。
2. **主路径重构**
   - Agent Tab 顶部增加“推荐下一步”区（如：发送首条任务、等待回传、停止并收敛）。
3. **事件分级显示**
   - 技术事件（debug）与业务事件（用户可读）分层。

验收：
- 关键动作反馈覆盖率 100%。
- 新手首次完成闭环任务成功率提升到 >80%。

---

## M2（第 3-4 周）：任务导向协作流（从“控制台”到“工作流”）

目标：让用户按任务组织多 agent，而不是按按钮操作。

范围：
1. **任务卡片模型**
   - 引入“任务卡（目标、负责人agent、状态、截止）”。
2. **一键分派**
   - 在任务卡里直接分配给某 agent tab。
3. **汇总视图**
   - 展示每个任务的最新结论、阻塞点、下一步。

验收：
- 单任务协作中，用户切页次数下降 30%。
- 任务状态可视化完整率 95%。

---

## M3（第 5-7 周）：可靠性与恢复（可持续运行）

目标：解决“能跑”到“敢用”的差距。

范围：
1. **恢复中心**
   - 启动时提示可恢复会话；支持按 agent/任务恢复。
2. **异常恢复策略**
   - spawn 失败、backend blocking、agent 无响应的统一重试策略。
3. **可观测性面板**
   - fallback 次数、延迟、失败率、活跃 agent 数趋势。

验收：
- 会话恢复成功率 >95%。
- 异常后 3 分钟内恢复率 >85%。

---

## M4（第 8-10 周）：高级体验与治理上线

目标：面向团队使用的治理能力。

范围：
1. **权限与安全策略**
   - 高风险动作确认模板（Kill/批量操作）。
2. **资源治理**
   - 并发上限、超时策略、自动降载。
3. **协作质量增强**
   - teammate 回传摘要标准化（结论/证据/风险/下一步）。

验收：
- 生产可用评审通过（产品/研发/测试/安全）。
- 回归通过率 ≥95%。

---

## 6. 关键功能优化建议（按模块）

## 6.1 Workbench 右栏（控制区）

- 增加“建议模式”提示：基于平台自动推荐 in-process/tmux。
- teammate 列表支持排序/筛选（运行中优先、异常优先）。
- 支持批量操作（批量停止、批量重启）。

## 6.2 Agent Tab（独立会话）

- 增加“会话状态胶囊”：空闲/执行中/等待/异常。
- 消息区支持 role 颜色 + 语义标签（指令、进展、异常）。
- 增加“导出会话摘要”。

## 6.3 Collaboration 页

- 提供“团队态势视图”：健康度、阻塞、吞吐。
- 增加从 Collaboration 直接发任务给指定 agent 的轻量入口。

## 6.4 事件时间线

- 默认展示“用户可读事件”，高级模式展示技术事件。
- 支持按 agentId、事件类型、严重级别过滤。

## 7. 指标埋点建议（必须做）

建议新增埋点事件：
- `ma_spawn_clicked / ma_spawn_succeeded / ma_spawn_failed`
- `ma_tab_created / ma_tab_closed / ma_tab_closed_with_stop|kill`
- `ma_blocking_shown / ma_setup_hint_clicked`
- `ma_session_restored / ma_session_restore_failed`
- `ma_message_sent / ma_message_delivery_failed`

并输出周报：
- 漏斗：创建协作 → 分配任务 → 首次回传 → 完成任务
- 失败 Top 原因（backend、权限、超时、用户中断）

---

## 8. 风险与依赖

主要风险：
1. 技术状态复杂导致 UI 逻辑膨胀。
2. tmux/iTerm2 平台差异影响一致性体验。
3. 事件噪声过高影响用户决策。

关键依赖：
- 多智能体后端能力（Phase 2 后半）
- 事件契约稳定性（`multiagent.events.ts`）
- 会话与记忆持久化结构冻结

---

## 9. 发布策略（建议）

- **灰度顺序**：
  1) 内部研发试用  
  2) 测试团队试用  
  3) 小范围真实任务场景试运行
- **版本策略**：
  - 每个里程碑独立 feature flag
  - 优先保障可回滚

## 10. 建议下一步（立即执行）

1. 先落 M1：统一反馈层 + 事件分级显示（投入小、收益大）。
2. 并行启动 M2 的任务卡原型（产品稿 + 可点击交互）。
