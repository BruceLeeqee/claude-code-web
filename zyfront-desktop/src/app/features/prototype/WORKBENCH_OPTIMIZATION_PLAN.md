# Workbench 对标标准原型优化计划

更新时间：2026-04-05
范围：`/workbench` 页面（含终端、资源树、任务面板、状态栏）
目标：与标准原型在**信息架构、交互行为、视觉层级**上达到高一致性，并保持可对接 `zyfront-core`。

---

## 一、当前对标结论（差异清单）

### A. 结构与布局
1. **顶部栏信息密度仍偏低**
   - 原型：含连接状态、内存、任务开关、搜索、设置，且视觉节奏更紧凑。
   - 当前：已有基础元素，但间距、字号、层次还不够“IDE化”。

2. **左侧资源树可用但拟真度不足**
   - 原型：目录/文件行高更紧，层级缩进更自然，选中态更明显。
   - 当前：功能完整（展开/读取），但视觉细节和交互反馈还不够贴近。

3. **中部终端与输入区接近，但“命令提示体验”仍可提升**
   - 原型：命令提示更像命令面板，输入反馈更连贯。
   - 当前：已支持补全、历史、NL 分流，但提示层样式与键盘交互还可强化。

4. **右侧任务面板内容是半静态**
   - 原型：状态、进度、标识有更强“实时感”。
   - 当前：UI 有雏形，但尚未完全绑定 coordinator 实时状态。

### B. 交互行为
1. **终端交互已是真 PowerShell（已达标）**
   - 已实现 pty 双向通信、粘贴复制、resize 同步。

2. **插件指令体系已接入但协议仍是轻量版**
   - `/plugin:list`、`/plugin:run`、`/superpowers:*` 已有入口。
   - 仍缺统一指令注册表（元数据、权限、参数校验、帮助文档自动生成）。

3. **自然语言与 shell 自动分流规则需要更稳健**
   - 当前通过中文/句式判断，需改为“可配置策略 + 明确优先级”。

### C. 视觉一致性
1. 字体、边框、阴影、间距仍有“接近但不一致”点。
2. 终端内容区、任务卡片、资源树 hover/active 的色彩语义需统一到一套 token。
3. 某些模块在高 DPI 下密度偏松，需做响应式密度档位。

---

## 二、优化目标（验收标准）

### 目标 G1：界面一致性
- 与标准原型对比，核心区域（顶栏/资源树/终端/任务面板）视觉差异控制在可接受范围（主观评估 ≥ 90%）。

### 目标 G2：交互完整性
- 输入框支持：历史、补全、命令面板、自然语言/插件/shell 路由。
- 终端支持：实时输出、复制粘贴、窗口缩放同步。

### 目标 G3：运行时联动
- 任务面板真实绑定 coordinator steps，状态变化可追踪。
- 插件指令与 `zyfront-core` tool/plugin/skill 系统形成可扩展映射。

---

## 三、分阶段实施计划

## Phase 1（P1）视觉精修：先做到“看起来一致”
优先级：高
工期：0.5~1 天

- [ ] 统一 UI token（颜色/阴影/圆角/字号/间距）
- [ ] 顶栏重排：状态信息右对齐、图标尺寸标准化
- [ ] 资源树行高、缩进、选中态、hover 态精修
- [ ] 输入区与命令提示层视觉升级（更像命令面板）
- [ ] 右侧任务卡片层级和状态色规范化

交付：视觉对比截图（当前 vs 原型）+ 样式 token 表

---

## Phase 2（P2）交互深化：先做到“用起来顺”
优先级：高
工期：1 天

- [ ] 命令提示支持键盘上下选择 + Enter 提交 + Esc 关闭
- [ ] 历史命令支持前缀过滤（如输入 `/pl` 后只遍历插件相关历史）
- [ ] 输入路由策略配置化：directive > shell > natural-language（可切换）
- [ ] 终端常用快捷键增强（Ctrl+L 清屏、Ctrl+K 清输入）
- [ ] 插件指令参数校验与错误提示统一

交付：交互测试清单（键盘路径）+ 演示录屏

---

## Phase 3（P3）运行时对接：先做到“状态是真实的”
优先级：最高
工期：1~1.5 天

- [ ] 任务面板绑定 `coordinator.getState().steps`
- [ ] 将“已启动/进行中/待处理”改为实时聚合结果
- [ ] `/status` 输出补全：mode/steps/toolCalls/sessionCost
- [ ] `/plugin:list` 改为读取 runtime 插件注册信息
- [ ] `/plugin:run` 对接工具执行日志（成功/失败/耗时）

交付：运行时联动验收（指令触发 -> 面板变化 -> 日志记录）

---

## Phase 4（P4）质量与稳定性
优先级：中
工期：0.5 天

- [ ] 终端会话生命周期检查（页面切换、销毁、重建）
- [ ] 异常恢复（pty 退出、IPC 失败、网络中断）
- [ ] 性能检查（大目录展开、长日志滚动、补全列表）
- [ ] 构建/类型/lint 全绿

交付：质量报告 + 回归清单

## 四、技术改造点（建议）

1. 新增 `command-router.service.ts`
   - 负责 directive/shell/nl 三路分发
   - 提供可测试的纯函数策略

2. 新增 `directive-registry.ts`
   - 统一管理命令元数据：名称、参数、帮助、执行器

3. 新增 `task-panel.facade.ts`
   - 从 `zyfront-core` 聚合任务状态并映射 UI

4. 样式层新增 token 文件
   - 例如：`prototype.tokens.scss`

---

## 五、风险与对策

1. **node-pty 在 Windows 兼容细节差异**
   - 对策：统一 PowerShell 启动参数，增加会话健康检查。

2. **自然语言/命令误判**
   - 对策：增加显式前缀（`!` 强制 shell，`?` 强制 AI）作为兜底。

3. **任务面板与 runtime 状态不同步**
   - 对策：统一由 facade 订阅单一状态源，避免双写。

- 指令补全命中率（手工测试） > 95%
- 任务面板状态刷新延迟 < 200ms
- 构建与 lint 100% 通过
