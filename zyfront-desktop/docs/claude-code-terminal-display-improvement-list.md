# Claude Code 终端展示改进列表

> 参考 `claude-code/restored-src/src` 的终端面板与思考块展示方式，对照当前 `zyfront-desktop` 的 `workbench.page.ts`、`command-router.service.ts`、`command-processing.service.ts`、`prompt-memory-builder.service.ts` 等实现，整理当前项目在终端展示方面的不足与可改进项。

---

## 1. 结论摘要

当前项目已经具备以下基础能力：

- xterm 主终端集成
- 思考块编号、marker、sessionStorage 恢复
- Ctrl+O / Ctrl+Shift+O 的思考块折叠展开逻辑
- 命令路由、执行、预处理的模块化拆分
- 提示词记忆层的读取与预算控制

但与 Claude Code 的终端展示相比，仍存在几个明显差距：

1. 终端宿主层没有像 Claude Code 那样独立、稳定、可切换的会话抽象。
2. 思考块展开仍偏“覆盖式/插入式”实现，布局稳定性不足。
3. 展开、收起、回放、恢复之间缺少统一状态机。
4. 滚动锚点与 buffer 行号强耦合，重绘后容易漂移。
5. 展示层与命令/记忆层联动弱，可解释性不足。
6. 历史回放与实时流的展示策略还没有完全统一。
7. 终端展示相关能力分散在 `workbench.page.ts`，页面职责过重。
8. 对长会话、热更新、切页恢复的稳定性仍可加强。

---

## 2. 现有实现对照

### 2.1 当前项目已有的能力

从当前代码可见，已有这些关键点：

- `thinkingBlocksById`：思考块元数据缓存
- `thinkingBlockMarkers`：定位锚点
- `mergeThinkingBlocksFromSession()`：sessionStorage 恢复
- `expandThinkingViaOverlay()`：展开逻辑
- `collapseThinkingInline()`：收起逻辑
- `disposeAllThinkingOverlays()`：统一清理
- `Ctrl+O` / `Ctrl+Shift+O`：快捷键支持
- `command-router.service.ts`：命令路由判断
- `command-processing.service.ts`：输入预处理与执行流
- `prompt-memory-builder.service.ts`：记忆读取与 prompt 拼装

### 2.2 与 Claude Code 的核心差异

Claude Code 恢复源码强调的是：

- 终端是持久会话宿主，不是一次性 UI 绘制
- 展开/折叠本质是状态切换，不是额外覆盖渲染
- 终端切换要保留 shell 状态与交互上下文
- 展开思考块时尽量让位、重排，而不是简单覆盖

当前项目虽然已经接近这个方向，但还没有完全做到“终端宿主稳定化 + 思考块状态机化 + 回放统一化”。

---

## 3. 当前项目在终端展示方面的不足清单

## 3.1 终端宿主层不足

### 问题 1：终端宿主仍过度依赖页面组件

当前终端核心逻辑大量集中在 `workbench.page.ts`，页面既负责：

- xterm 生命周期
- 终端输出
- 思考块状态
- 快捷键处理
- sessionStorage 恢复
- 滚动处理
- 命令交互

这使得页面职责过重，后续很容易继续膨胀。

### 问题 2：缺少独立的 terminal host / session 抽象

Claude Code 的终端面板是一个明确的宿主对象，支持进入、退出、保持会话、恢复状态。当前项目更像是“页面里挂了一个 xterm”，缺少统一宿主服务。

### 问题 3：终端切换与恢复的状态边界不够清晰

当前虽然有路由复用和会话恢复逻辑，但缺少类似 Claude Code 的“面板状态机”，导致长会话切页、热更新、刷新后的恢复稳定性仍有提升空间。

---

## 3.2 思考块展示不足

### 问题 4：展开/收起主路径仍偏覆盖式实现

当前使用：

- `expandThinkingViaOverlay()`
- `insertVisualRowsAfterFold()`
- `collapseThinkingInline()`

这套逻辑能工作，但仍然比较依赖 buffer 行号、viewport 和临时空行的同步状态。

### 问题 5：结构级位移不够统一

Claude Code 的思路更接近“先让位，再展示”，而当前实现更像“先插空，再写回”。两者接近，但后者对滚动和重绘更敏感。

### 问题 6：展开态与折叠态没有完全作为同一状态对象处理

虽然已经有 `thinkingBlocksById` 和 marker，但展开、折叠、全部展开、全部收起仍有多个分支逻辑，缺少更统一的 block state machine。

### 问题 7：思考块恢复逻辑偏散

`mergeThinkingBlocksFromSession()` 能恢复 sessionStorage，但对热更新、重绘、重入、重复注册的统一治理还不够强。

---

## 3.3 锚点与滚动不足

### 问题 8：过度依赖 buffer 行号

当前通过 marker + buffer 行号找块，但展开 / 收起 / 清屏 / 回放后，buffer 行号很容易失去稳定性。

### 问题 9：滚动锚定与展开行为联动不够统一

当前已有避免滚到底的注释和策略，但整体上还没有一个明确的“滚动锚点协调器”，来统一处理：

- 展开后滚动保持
- 收起后锚点回退
- 历史定位
- 最近思考块聚焦

### 问题 10：重绘与可见区域管理不足

思考块扩展后，视觉上可能发生后续内容位移、视口跳动或定位偏差，缺少专门的布局协调层。

---

## 3.4 历史回放与实时流不足

### 问题 11：实时流和历史回放共用代码，但未完全共用状态模型

当前实现中，实时流、回放、恢复、清理之间仍有多处在 `workbench.page.ts` 内部直接处理，缺少独立 replay coordinator。

### 问题 12：回放模式与完整模式未完全分离

Claude Code 的展示更强调“折叠默认、需要时展开”。当前项目历史回放场景下的展示策略还可以更明确地拆成：

- 紧凑回放
- 完整回放
- 交互展开回放

### 问题 13：回放状态恢复的语义还不够统一

在刷新、切页、重新加载后，思考块恢复逻辑依赖 sessionStorage 与当前内存状态的合并，仍有边界情况可能导致编号、位置、展开态不一致。

---

## 3.5 命令与终端展示联动不足

### 问题 14：路由层与展示层耦合度仍偏高

`command-router.service.ts`、`command-processing.service.ts` 已经模块化，但终端展示逻辑仍在页面层直接消费执行结果，缺少专门的展示协调层。

### 问题 15：命令结果缺少统一的展示语义

当前 directive、shell、natural 的结果虽然分流了，但在终端展示中还没有完全统一成：

- 提示型
- 成功型
- 错误型
- 系统型

这样的稳定展示协议。

### 问题 16：缺少“如何显示”的中间层

也就是说，现在更多是“路由到哪里、执行什么”，但没有一层专门决定“终端怎么渲染这一条结果”。

---

## 3.6 记忆与终端展示联动不足

### 问题 17：prompt 记忆构建与终端展示还没有形成闭环

`prompt-memory-builder.service.ts` 已经能拼装记忆层，但终端展示里看不到记忆层构建结果的可解释摘要。

### 问题 18：缺少“当前会话上下文面板”

Claude Code 风格更强调上下文稳定可见。当前项目还缺少一个统一的会话上下文视图，展示：

- 当前轮次
- 当前思考块
- 当前 prompt 记忆层
- 当前路由结果
- 当前展开状态

### 问题 19：展示层无法反向解释记忆层

当记忆没有更新或被截断时，终端层并不能清晰解释为什么看不到某些信息。

---

## 3.7 稳定性与工程化不足

### 问题 20：`workbench.page.ts` 过于庞大

终端展示、输入处理、状态恢复、思考块管理、快捷键、输出渲染都堆在一个大页面中，后续改动风险高。

### 问题 21：缺少独立的 terminal debugging / regression 组件

当前没有看到专门用于：

- 记录思考块状态变化
- 记录展开/收起行为
- 记录滚动锚点
- 记录回放恢复

的调试服务。

### 问题 22：长会话和热更新场景仍容易积累复杂状态

虽然有清理和恢复，但没形成完整状态机的情况下，长会话很容易出现：

- 已展开块残留
- marker 失效
- sessionStorage 与内存不同步
- 视图跳动

---

## 4. 改进列表

下面给出按优先级排序的改进项。

### P0 - 必做

1. **抽出独立终端宿主服务**
   - 新建 `terminal-host.service.ts` 或同类抽象
   - 从 `workbench.page.ts` 中剥离终端宿主生命周期

2. **抽出思考块管理器**
   - 新建 `thinking-block-manager.service.ts`
   - 统一注册、查找、恢复、展开、收起、清理

3. **统一展开/收起状态机**
   - 将 `Ctrl+O`、`Ctrl+Shift+O` 的逻辑收敛到单一状态层
   - 让折叠态、展开态、全部展开态有明确迁移关系

4. **统一锚点服务**
   - 新建 `terminal-anchor.service.ts`
   - 统一 marker、buffer 行号、最近块定位、滚动恢复

5. **把覆盖式展开改为块级重绘主路径**
   - 减少直接操作 viewport 的次数
   - 优先使用统一 renderer 输出折叠/展开态

### P1 - 强烈建议

6. **抽出终端渲染协调层**
   - 新建 `terminal-block-renderer.service.ts`
   - 统一 ANSI、占位、摘要、展开渲染

7. **抽出历史回放协调层**
   - 新建 `session-replay.service.ts`
   - 统一实时流与回放流的 block 数据模型

8. **把记忆构建结果暴露到终端调试视图**
   - 让 prompt memory 的构建报告可见
   - 解释为什么某层被截断或未显示

9. **统一命令结果展示语义**
   - directive / shell / natural 的输出分类明确化
   - 在终端里统一用同一套展示协议

10. **提升 sessionStorage 恢复鲁棒性**
    - 恢复时校验版本、编号、状态、marker 一致性
    - 避免重复恢复和错误合并

### P2 - 进一步优化

11. **增加终端调试面板**
    - 展示最近思考块状态变化
    - 展示滚动锚点和展开记录
    - 展示回放恢复结果

12. **拆分 `workbench.page.ts` 的职责**
    - 页面只保留视图与入口
    - 终端、思考块、回放、快捷键都下沉到服务层

13. **增加长会话回归测试**
    - 连续多轮思考块展开/收起
    - 多次切页恢复
    - 热更新后恢复
    - 大量 block 场景下的性能测试

14. **补齐“当前会话上下文面板”**
    - 显示当前 turn、块数、展开态、路由结果、记忆层状态

15. **统一系统状态报告**
    - 把终端、记忆、路由、执行的状态汇总到一个调试入口

---

## 5. 推荐的改造顺序

建议按下面顺序推进：

1. 终端宿主抽象
2. 思考块管理器抽象
3. 状态机与锚点服务统一
4. 渲染协调层拆分
5. 回放协调层拆分
6. 记忆与终端的调试视图联动
7. 页面大组件职责下沉
8. 长会话与热更新回归测试

---

## 6. 总结

当前项目在终端展示方面已经有不错的基础，但和 Claude Code 相比，最大的差距不是“能不能显示”，而是：

- 是否有稳定的终端宿主
- 是否有统一的思考块状态机
- 是否能保证展开/收起不破坏结构
- 是否能在回放、恢复、热更新后仍然稳定
- 是否能把展示、命令、记忆三者真正串成闭环

因此，后续优化的方向应该从“补功能”转为“收敛状态、拆分职责、统一渲染、增强恢复”。
