## 2. 参考源码的核心设计思想

### 2.1 统一输入处理链路
参考源码不是简单地“识别一个命令前缀”，而是采用一条完整的输入处理链路：

1. 输入标准化
2. 特殊来源识别
3. 附件/图片/桥接输入处理
4. slash / bash / natural 分流
5. hooks 与额外上下文处理
6. 进入具体执行器

这一点决定了命令系统不是单独一个路由器，而是输入总线的一部分。

### 2.2 注册式命令系统
参考源码中的命令体系以“注册、解析、分发、执行”为主，而不是散落在 UI 中的条件分支。

表现为：
- 命令元数据统一登记
- 命令与别名统一查找
- 帮助信息、可见性、可执行条件集中管理
- 新命令接入成本低

### 2.3 安全优先与上下文隔离
参考源码对以下情况都有明确区分：

- 本地用户输入
- 桥接输入 / 远程控制输入
- 系统生成提示
- 不同模式下的命令可用性

这意味着命令系统不仅要“能识别”，还必须“知道什么时候不能执行”。

## 4. 对标差距清单

以下清单按“影响程度 + 复刻优先级”排序。

---

### 差距 1：输入处理链路不完整

#### 参考源码能力
参考源码将输入处理拆成多个步骤：
- 输入预处理
- 图像/附件处理
- slash 命令识别
- bash 命令识别
- 自然语言路由
- hooks / 额外上下文处理

#### 当前项目现状
当前主要依赖 `CommandRouterService` 做“输入类别判断”，尚未形成完整的预处理流水线。

#### 差距影响
- 复杂输入场景容易误判
- 后续扩展桥接消息、附件输入、系统注入输入会受限
- 路由与执行耦合偏高

#### 建议补齐
- 增加输入预处理层
- 将“判定”和“执行”分离
- 为后续附件、桥接、系统提示留统一入口

---

### 差距 2：命令解析层过轻

#### 参考源码能力
参考源码有独立的 slash 命令解析与命令查找过程：
- 解析命令名
- 提取参数
- 查找命令定义
- 判断命令可执行性
- 对未知命令做回退或友好提示

#### 当前项目现状
当前更偏向字符串 heuristics，解析层能力较轻。

#### 差距影响
- `plugin:*`、带冒号命令、复杂参数命令不易统一支持
- 别名、分组、显示说明等信息利用不足
- 命令输入与命令定义之间的关系不够严谨

#### 建议补齐
- 新增独立 `DirectiveParser`
- 支持 name / alias / group / usage / template / mode 条件
- 输出结构化解析结果而非仅类别判断

---

### 差距 3：命令定义元数据不够完整

#### 参考源码能力
参考源码中的命令通常具备较完整的元信息，例如：
- 可见性
- 使用条件
- 分组
- 别名
- 模式限制
- 帮助内容
- 入口类型

#### 当前项目现状
当前 `DirectiveDefinition` 更接近轻量配置，仍偏“展示层定义”。

#### 差距影响
- 不能很好支撑大型命令系统
- 帮助菜单、模式禁用、平台兼容等能力不足
- 新命令扩展需要改多处逻辑

#### 建议补齐
- 统一扩展 `DirectiveDefinition`
- 让 registry 承担更多“命令元数据中心”职责
- 把帮助/模板/可见性/启用条件都纳入定义

---

### 差距 4：shell / directive / natural 的边界不稳定

#### 参考源码能力
参考源码明确区分：
- slash 命令
- bash 命令
- 自然语言 prompt

并且对“桥接输入”“系统输入”“特殊模式”有额外保护。

#### 当前项目现状
当前路由虽可工作，但在以下场景仍可能边界不稳：
- 中文自然语言混入 shell 关键词
- 路径、管道、重定向、变量等复合 shell 片段
- `/` 开头但并非真正命令的输入

#### 差距影响
- 用户输入体验不稳定
- 容易误触发错误路由
- 后续集成更复杂命令时风险上升

#### 建议补齐
- 提高自然语言优先级识别能力
- 增强 shell 特征判别
- 对歧义输入增加解释字段

---

### 差距 5：缺少统一执行适配层

#### 参考源码能力
参考源码将不同类型输入分发给不同处理器：
- slash command executor
- bash command executor
- text prompt processor

#### 当前项目现状
当前 workbench 层还偏向“路由后直接处理”，执行边界不够统一。

#### 差距影响
- 错误回显格式不统一
- 执行日志分散
- 以后增加新类型输入时容易重复实现

#### 建议补齐
- 引入 `CommandExecutionAdapter`
- 统一成功、失败、未知、不可执行等回显格式
- 路由结果统一进入适配层

---

### 差距 6：缺少“可解释路由结果”

#### 参考源码能力
参考源码在复杂路径中会保留足够多的上下文来解释为什么这样路由、为什么不执行。

#### 当前项目现状
当前更多是“判定结果”，可解释性不足。

#### 差距影响
- 调试困难
- 用户无法理解为什么输入被识别为某一类
- 容易把问题归咎于命令系统本身

#### 建议补齐
- 路由结果增加 `reason` / `signals` / `confidence`
- 对歧义输入输出解释信息
- 便于将来做遥测和诊断

---

### 差距 7：特殊输入来源隔离不够

#### 参考源码能力
参考源码对桥接输入、系统生成提示、远程控制消息有明显隔离。

#### 当前项目现状
当前虽然有部分系统上下文概念，但对特殊来源输入的隔离还不够完整。

#### 差距影响
- 系统输入可能误触发本地命令
- 远程控制输入的安全边界不够清晰
- 后续多端同步场景容易混淆

#### 建议补齐
- 为输入源增加显式标记
- 在预处理阶段完成源隔离
- 针对不同来源定义不同路由策略

---

### 差距 8：帮助体系与命令元数据输出不完整

#### 参考源码能力
参考源码的命令元数据不仅用于执行，也用于：
- help 展示
- 菜单生成
- 可见性控制
- 模式筛选
- 平台筛选

#### 当前项目现状
当前已有 template / usage，但还不足以支持完整帮助体系。

#### 差距影响
- `/help` 可扩展性不足
- 指令列表生成能力有限
- UI 展示层与命令定义层耦合偏高

#### 建议补齐
- 增加 `group`、`aliases`、`visibleInHelp`、`enabledWhen`
- Registry 负责统一暴露给 UI
- 帮助与执行共享同一份定义源

---

### 差距 9：复杂场景回退策略不够细

#### 参考源码能力
参考源码对未知、不可执行、条件不满足等情形通常不会直接“断链”，而会采取更平滑的处理。

#### 当前项目现状
当前对于未识别命令、冲突命令、边界输入的回退策略仍可优化。

#### 差距影响
- 用户体验不够柔和
- 容易出现“输入被错误拦截”或“无提示静默失败”

#### 建议补齐
- 未知命令统一回退到自然语言或明确提示
- 不可执行命令给出原因与替代建议
- 邻近命令可提供纠错候选

---

### 差距 10：缺少面向未来的扩展架构

#### 参考源码能力
参考源码的命令体系天然适合持续扩展：
- 新命令可通过注册方式接入
- 新模式可通过元信息控制
- 新执行器可通过适配层接入

#### 当前项目现状
当前系统已经具备雏形，但如果继续靠增量判断分支，后续会越来越难维护。

#### 差距影响
- 命令数量一多，逻辑会快速膨胀
- 规则冲突概率升高
- 复刻 Claude Code 风格会受限

#### 建议补齐
- 明确四层架构：预处理 / 解析 / 路由 / 执行
- 统一命令注册中心
- 统一返回结构与错误处理

## 6. 建议的最终目标架构

建议将当前命令系统升级为以下结构：

```text
用户输入
  -> Command Preprocessor
  -> Input Source Guard
  -> Directive Parser / Shell Detector / Natural Classifier
  -> Command Router
  -> Execution Adapter
  -> Concrete Executor
  -> Unified Result Formatter
```

其中：

- `Command Preprocessor`：负责标准化输入
- `Input Source Guard`：负责隔离桥接、系统生成等特殊来源
- `Directive Parser`：负责 slash 命令解析
- `Shell Detector`：负责 bash / shell 检测
- `Natural Classifier`：负责自然语言判断
- `Command Router`：负责路由决策
- `Execution Adapter`：负责执行器分发
- `Unified Result Formatter`：负责统一回显与错误格式

## 7. 推荐的落地改造顺序

### Phase 1：先补架构边界
- 拆出预处理层
- 拆出解析层
- 拆出执行适配层

### Phase 2：再补元数据能力
- 扩展 `DirectiveDefinition`
- 增加别名、分组、可见性、模式约束
- 统一 help 输出

### Phase 3：再补判定与回退
- 强化 shell / natural 边界
- 增加 unknown 与 unavailable 回退
- 增加路由解释信息

### Phase 4：最后做体验与治理
- 统一错误提示
- 统一日志与遥测
- 为新增命令留注册入口

---

## 8. 结论

当前 `zyfront-desktop` 的命令系统已经具备一个可用的基础框架，但与 `claude-code/restored-src/src` 的真实命令系统相比，核心差距不在“能不能识别命令”，而在于：

1. 是否拥有完整的输入处理流水线；
2. 是否具备注册式、可扩展、可解释的命令解析体系；
3. 是否能稳定处理复杂输入边界与特殊来源隔离；
4. 是否有统一执行适配与统一回显格式；
5. 是否为未来命令膨胀预留了结构化治理能力。

如果后续要真正实现“对标 Claude Code 设计思想”的命令系统，建议优先补齐 **预处理、解析、适配、元数据** 四个层面的基础设施，而不是继续在单一路由器上堆规则。
