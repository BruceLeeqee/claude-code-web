# 多智能体无人值守开发团队重构设计文档

> 目标：将 `loop` 从“单体执行器”升级为“多智能体自治编排器”，支持无人值守完成大型开发任务。`loop` 不仅可启动 `/mode-dev` 团队开发，也可以承载规划、验证、研究、修复、总结等其他任务。系统必须先完成需求文档与设计文档的确认，再进入开发执行；同时把 `/mode-dev` 整合为 loop 的标准开发团队入口。`loop` 默认采用类似 `tmux` 的后台持续运行模式，不中断执行，直到任务完成、阻塞、暂停或达到收敛条件。本文档用于指导后续自动生成需求、设计、执行、验证与总结文档。

---

## 1. 目标与边界

### 1.1 目标
- 支持复杂任务的持续推进，不中断运行
- 支持需求澄清、方案设计、开发执行、测试验证、失败修复、收敛发布
- 支持编译、UI、接口、数据、终端、沙箱浏览器等多维验证
- 支持每一步自动生成文档与状态记录
- 支持通过命令把任务派发给不同团队完成
- 支持开发项目默认进入 `/mode-dev` 团队模式
- 支持 `loop` 作为多任务总入口，不仅限于开发任务
- 支持 `loop` 默认以后台持续运行模式执行，不阻断当前会话

### 1.2 边界
- 对开发类任务，必须先完成需求文档与设计文档，再允许进入编码阶段
- 高风险操作必须保留门禁：自动提交、自动推送、自动发布、批量删除、跨模块重构
- `loop` 是自治编排器，不是单纯的死循环执行器

---

## 2. 设计原则

### 2.1 先需求，后设计，再开发
这是最高优先级原则。

对于任何开发类任务，系统必须按如下顺序推进：
1. 需求澄清
2. 需求文档确认
3. 设计文档确认
4. 开发实施
5. 编译与验证
6. 修复与重试
7. 收敛与交付

在设计文档未确认前，系统不得进入实际代码修改阶段。

### 2.2 文档驱动
系统的每一步都应该有对应文档：
- 需求文档
- 设计文档
- 执行文档
- 验证文档
- 修复文档
- 总结文档

### 2.3 团队驱动
loop 不直接假设单个模型可以完成全部任务，而是通过多智能体团队分工协作完成。

### 2.4 后台持续运行驱动
loop 默认采用后台持续执行策略，类似 `tmux` 中的长期会话：
- 命令发出后立即返回控制权
- loop 在后台持续推进任务
- 通过状态面板、日志与文档持续可见
- 遇到阻塞时保留上下文并等待下一次调度或用户介入

### 2.5 证据驱动
所有验证结论都应基于证据：
- 编译输出
- 测试输出
- 截图
- DOM 状态
- 网络请求
- 数据校验结果
- 终端执行结果

### 2.6 可中断与可回退
任何时候用户都可以：
- 暂停
- 修改目标
- 接管
- 回退修复
- 重新规划

## 3. 为什么要重构开发团队

当前的开发团队更像是“通用协作容器”，还不够适合无人值守的大任务。要实现真正的自治开发，需要让团队具备以下能力：

- 明确角色边界
- 可按任务类型动态组队
- 每个角色有标准输入/输出
- 每个角色都能产出文档和证据
- 团队内部可协作、可轮转、可追责
- loop 统一调度，而不是单点驱动

参考成熟自动化方案的共性经验，可归纳为：
- Planner 先做结构化拆解
- Developer 负责实现
- Tester 负责详细测试案例
- Verifier 负责结果判定
- Repairer 负责失败修复
- Orchestrator 负责状态与资源调度

---

## 4. 推荐团队组成

### 4.1 核心团队角色

#### 4.1.1 总编排器 Orchestrator
职责：
- 接收命令
- 判断任务类型
- 创建或选择团队
- 分配子任务
- 推进状态流转
- 控制重试与收敛
- 维护后台会话与状态可见性

#### 4.1.2 需求分析师 Requirement Analyst
职责：
- 与用户进行需求澄清
- 输出需求文档
- 整理范围、约束、成功标准
- 标记待确认问题
- 识别是否属于开发任务

#### 4.1.3 方案设计师 Solution Architect
职责：
- 根据需求文档输出设计文档
- 给出架构、接口、状态模型、验证方案
- 识别技术风险
- 确认何时允许进入开发

#### 4.1.4 开发执行者 Implementer
职责：
- 修改代码
- 生成补丁
- 处理依赖关系
- 记录文件变更
- 仅在需求/设计确认后进入

#### 4.1.5 测试工程师 Tester
职责：
- 编写详细测试案例
- 运行单测、集成测试、smoke 测试
- 生成测试报告
- 为 sandbox/browser 运行提供测试脚本

#### 4.1.6 验证工程师 Verifier
职责：
- 负责编译、UI、接口、数据验证
- 读取截图、DOM、console、network、stdout/stderr
- 给出继续 / 修复 / 暂停 / 收敛建议
- 判断是否满足发布门槛

#### 4.1.7 修复工程师 Repairer
职责：
- 根据失败输出定位问题
- 生成修复计划
- 应用结构化补丁
- 重跑最小验证集
- 处理重复失败的升级策略

#### 4.1.8 发布门禁 Reviewer
职责：
- 审查是否满足发布条件
- 检查高风险动作门禁
- 决定是否进入 ready_for_release
- 确认是否需要用户审阅后再收敛

#### 4.1.9 后台守护 Runner
职责：
- 维护长期后台运行
- 管理 tmux-like 会话
- 监控任务是否卡死
- 在必要时自动恢复执行

## 5. 推荐团队结构

### 5.1 开发任务团队
建议 `/mode-dev` 默认创建如下团队：
- `orchestrator`
- `requirement-analyst`
- `solution-architect`
- `implementer`
- `tester`
- `verifier`
- `repairer`
- `reviewer`
- `runner`

### 5.2 非开发任务团队
`loop` 还应该支持其他任务类型团队：
- `plan-team`：负责规划类任务
- `research-team`：负责需求研究与方案比较类任务
- `qa-team`：负责测试验证类任务
- `release-team`：负责发布前收敛类任务
- `ops-team`：负责终端、沙箱、运行时观测类任务
- `docs-team`：负责文档生成与整理类任务

### 5.3 轮转机制
当某个角色产出不足或失败时，loop 可以：
- 重新分配同角色
- 调用上一级角色复审
- 切换到更高抽象层重新规划
- 将任务转交给另一个更合适的团队

## 6. `/mode-dev` 与 loop 的关系

### 6.1 定位
`/mode-dev` 是开发项目的默认团队模式入口。它不是独立系统，而是 loop 的一个标准执行场景。

### 6.2 流程
开发类任务进入 loop 后，推荐流程如下：
1. 识别为开发任务
2. 进入需求澄清阶段
3. 输出需求文档
4. 输出设计文档
5. 确认设计细节
6. 调用 `/mode-dev` 创建团队
7. 团队分工执行
8. 编译、浏览器、接口、数据验证
9. 失败修复与重试
10. 收敛交付

非开发类任务也可以通过 loop 进入对应团队，但默认仍采用后台持续运行模式，不要求人工持续盯守。

### 6.3 好处
- 统一团队入口
- 避免单个模型承担全部职责
- 更适合长期无人值守运行
- 更容易追踪责任与状态
- 支持开发与非开发任务统一编排

## 7. 三类核心文档体系

> 这里的“三分文档”建议理解为：需求文档、设计文档、执行/验证文档三大类。系统应优先自动生成这三类文档，并在任务推进时持续更新。

### 7.1 需求文档
用于回答：
- 要做什么
- 为什么做
- 范围是什么
- 约束是什么
- 成功标准是什么

### 7.2 设计文档
用于回答：
- 怎么做
- 模块如何划分
- 状态如何流转
- 接口如何定义
- 如何验证

### 7.3 执行/验证文档
用于回答：
- 实际做了什么
- 哪些文件改了
- 验证是否通过
- 失败原因是什么
- 下一步怎么继续

---

## 8. 文档自动生成策略

### 8.1 自动生成原则
后续系统应自动生成以下文档：
- 需求文档
- 设计文档
- 每步执行记录
- 验证记录
- 失败分析
- 总结报告

### 8.2 生成时机
- 任务进入前：生成需求文档草案
- 需求确认后：生成设计文档
- 每次执行后：生成执行记录
- 每次验证后：生成验证结果
- 发生失败后：生成失败分析
- 任务结束时：生成总结报告

### 8.3 存储建议
每个任务可以使用如下目录结构：
```text
/docs/loop/<taskId>/
  requirements.md
  design.md
  execution/
    step-001.md
    step-002.md
  verification/
    check-001.md
  failures/
    failure-001.md
  summary.md
```

## 9. 命令驱动的任务分派

### 9.1 建议命令
- `/loop <目标>`：启动自治流程
- `/mode-dev`：进入开发团队模式
- `/task team=<teamName> objective=<objective>`：指定团队完成任务
- `/task role=<role> objective=<objective>`：指定角色完成任务
- `/task plan <objective>`：仅规划，不执行
- `/task verify <objective>`：仅验证
- `/task team=ops objective=启动沙箱浏览器并检查页面`
- `/task team=docs objective=生成需求/设计文档`

### 9.2 命令路由规则
1. 先解析命令
2. 再识别任务类型
3. 再选择团队
4. 再选择执行阶段
5. 再决定是否需要用户确认
6. 再决定是否后台挂起或持续执行

### 9.3 例子
- `/task team=dev objective=实现登录页`
- `/task team=qa objective=补齐登录页测试`
- `/task team=review objective=检查发布门禁`
- `/task team=ops objective=执行页面 UI 验证`
- `/task team=docs objective=生成需求和设计文档`

## 10. 详细执行流程

### 10.1 开发任务标准流程
1. 用户提出目标
2. 需求分析师整理需求
3. 需求文档确认
4. 方案设计师生成设计文档
5. 设计文档确认
6. Orchestrator 调用 `/mode-dev`
7. 团队执行开发
8. Tester 编写详细测试案例
9. Verifier 执行编译 / UI / 接口 / 数据验证
10. Repairer 修复失败项
11. 重新验证
12. Reviewer 判断是否收敛
13. 产出总结文档
14. 任务结束后保留后台会话与完整文档轨迹

### 10.2 非开发任务流程
- 直接派发到对应团队
- 仍需文档、状态、证据

## 11. 验证矩阵

### 11.1 必要维度
- 编译 / typecheck / lint
- UI 效果与交互
- 后端接口
- 数据一致性
- 终端命令结果

### 11.2 验证标准
每个验证项必须返回：
- 是否通过
- 错误
- 警告
- 阻塞项
- 证据
- 建议动作

## 13. 成功标准
如果系统能够：
- 先需求、后设计、再开发
- 自动创建/调度开发团队
- 自动完成编译、页面、接口、数据验证
- 自动生成每一步文档
- 自动重试并收敛
- 长时间无人值守运行复杂任务
- `loop` 默认以后台持续运行模式执行，不阻断当前会话
- `loop` 可派发开发、研究、文档、验证、运维等多种任务

则说明设计方向正确。
