## 二、需优化的点

### 2.1 🔴 P0 — 功能缺失

#### O-01: `/task` 命令未被接入
`LoopTaskRouterService.parseTaskCommand()` 已实现，但 **没有任何调用方**。DebugCommandService / 命令注册表中均未路由 `/task`。

**影响**: 用户无法通过 `/task` 命令派发任务给指定团队，文档第 6 章承诺的功能不可用。

**修复**: 在 `DebugCommandService` 或 workbench 的命令注册中增加 `/task` 路由。

---

#### O-02: 5 个服务被注入但从未被调用
`LoopExecutorService` 注入了 10 个服务，但以下 5 个从未在执行流中被调用：

| 服务 | 注入行 | 实际使用 |
|------|--------|----------|
| `LoopTaskRouterService` | L6 | ❌ 从未调用 |
| `LoopTeamManagerService` | L7 | ❌ 从未调用 |
| `LoopSandboxRunnerService` | L8 | ❌ 从未调用 |
| `LoopTerminalSandboxService` | L9 | ❌ 从未调用（executor 自有 `execCommand()`） |
| `LoopReleaseGateService` | L10 | ❌ 从未调用 |

**影响**: 团队编排、沙箱浏览器验证、发布门禁等功能虽已编码但未接入主流程，等于"死代码"。

**修复**: 在 `executeStep()` 的各 step.type 分支中调用对应服务：
- `implementation` → `teamManager.assignSteps()` 获取指派信息
- `verification` → `sandboxRunner.openPage()` 做 UI 验证 + `terminalSandbox.runVerificationMatrix()` 替换手动 `execCommand`
- `release_check` → `releaseGate.checkReadiness()`
- `summary` → `releaseGate.checkGitCommitGate()` / `checkReleaseGate()`

---

#### O-03: `implementation` 步骤只写 marker 文件
```typescript
// loop-executor.service.ts L143-158
if (step.type === 'implementation') {
  const targetFile = '02-AGENT-MEMORY/01-Short-Term/loop-executor-marker.txt';
  const payload = [
    'TODO: replace this marker with a real code change patch generated by the loop executor.',
  ].join('\n');
  // ...写入 marker，没有真实代码修改
}
```

**影响**: 开发任务的核心步骤（代码修改）是空操作。Loop 目前只能做验证和修复，不能做实现。

**修复**: 需要 AI Agent 对接。当前可做的改进：
1. 将 `implementation` 步骤标记为 `requires-agent` 状态
2. 暂停 loop，等待外部 AI Agent 返回代码修改结果
3. Agent 修改完成后恢复 loop 进入验证阶段

---

#### O-04: 没有 loop 控制命令

| 缺失命令 | 作用 |
|----------|------|
| `/loop status` | 查看当前 loop 状态 |
| `/loop stop` | 手动停止运行中的 loop |
| `/loop resume` | 恢复暂停的 loop |
| `/loop step` | 单步执行一轮 |

**影响**: 用户启动 loop 后无法干预，只能等待它自行收敛或达到上限。

**修复**: 扩展 `loop-command-parser.ts` 支持 `/loop <subcommand>` 语法，在 `LoopCommandService` 中增加对应方法。

---

### 2.2 🟡 P1 — 逻辑缺陷

#### O-05: `LoopCommandService.verify()` 与 `LoopVerifierService.verify()` 重复
`LoopCommandService.runCycle()` 内部调用自己的 `verify()` 方法（L86），逻辑与 `LoopVerifierService.verify()` 高度重复但更简陋：
- 没有 `detectConsecutiveBlockers()`
- 没有 `detectNoProgress()`
- 没有 `determineRecommendation()` 修复判定规则
- recommendation 逻辑只有 `repair` / `stop` / `continue`，缺少 `pause` / `release`

**影响**: 如果有人直接调用 `LoopCommandService.runCycle()`，会走一条更弱的验证路径。

**修复**: `LoopCommandService.runCycle()` 应委托给 `LoopVerifierService`，或标记为 `@deprecated` 让调用方统一使用 `LoopExecutorService.runOnce()`。

---

#### O-06: `LoopCommandService` 和 `LoopTeamManagerService` 重复 `buildTeamMembers()` / `defaultTeamName()`
两处代码完全相同：

| 方法 | LoopCommandService | LoopTeamManagerService |
|------|-------------------|----------------------|
| `buildTeamMembers()` | L412-423 | L96-118 |
| `defaultTeamName()` | L405-410 | L88-94 |

**修复**: 删除 `LoopCommandService` 中的重复代码，改为注入 `LoopTeamManagerService`。

---

#### O-07: `inferTaskType()` 重复三处
1. `loop-command-parser.ts` L65-73
2. `LoopTaskRouterService.inferTaskType()` L153-161
3. 隐式在 `LoopCommandService.createInitialState()` 中

**修复**: 统一为 `LoopTaskRouterService.inferTaskType()`，其他地方引用。

---

#### O-08: `LoopState.stage` 与 `LoopState.phase` 冗余
```typescript
// loop-command.types.ts L74-75
stage: LoopPhase;
phase: LoopPhase;
```
两个字段类型相同，在 `createInitialState()` 中被赋相同值，在 `executeStep()` 中也同步更新。

**修复**: 只保留一个字段（推荐 `phase`），全局替换。

---

#### O-09: `LoopState.stepDocs` / `fileChanges` / `memoryRefs` 无界增长
`validationHistory` 和 `toolHistory` 有 `.slice(-50)` / `.slice(-100)` 限制，但以下字段无上限：
- `stepDocs[]` — 每轮 +1~2，12 轮 = ~24 项
- `fileChanges[]` — 每轮 +N，无上限
- `blockedReasons[]` — 只增不减（即使后续验证通过，旧的 blocker 仍保留）
- `artifacts[]` — 只增不减

**修复**: 对 `stepDocs` 和 `fileChanges` 也加 `.slice(-N)` 限制；`blockedReasons` 在验证通过时清空。

---

#### O-10: `LoopPatchEngineService.consecutiveFailures` 是内存状态
```typescript
// loop-patch-engine.service.ts L57
private consecutiveFailures = new Map<string, number>();
```
页面刷新后丢失，可能导致本应升级为 blocked 的错误重新进入 repair 循环。

**修复**: 将失败计数持久化到 `LoopState` 或 localStorage。

---

#### O-11: Dashboard / ArtifactStore 写入错误被静默吞掉
```typescript
// loop-executor.service.ts L440, L444
this.dashboard.writeDashboardToDisk(dashboardVm).catch(() => {});
this.artifactStore.writeIndexToDisk(index).catch(() => {});
```

**修复**: 至少 `console.warn` 记录失败原因，或写入 `LoopState.lastError`。

---

#### O-12: 模板路径不匹配
```typescript
// loop-doc-writer.service.ts L82
const templatePath = `docs/loop-templates/${fileName}`;
```
但 `window.zytrader.fs.read()` 的 `scope: 'vault'` 通常指向 `02-AGENT-MEMORY/` 目录，`docs/loop-templates/` 可能不可读。

**修复**: 将模板文件复制到 vault scope 内，或改用 `scope: 'workspace'`。

### 2.3 🟢 P2 — 代码质量 & 可维护性

#### O-13: `window.zytrader` 无类型声明
```typescript
declare const window: Window & typeof globalThis;
// 然后直接 window.zytrader.terminal.exec / window.zytrader.fs.write / window.zytrader.browser
```
所有 zytrader API 调用都是 `any` 类型，无编译期检查。

**修复**: 创建 `zytrader.d.ts` 类型声明文件。

---

#### O-14: 终端命令硬编码 `cmd.exe /c`
```typescript
// loop-executor.service.ts L162, L168, L377 等
'cmd.exe /c npm test -- --runInBand'
```
macOS / Linux 环境无法运行。

**修复**: 根据 `navigator.platform` 或 `process.platform` 动态选择命令前缀。

---

#### O-15: `execCommand()` 无超时
```typescript
// loop-executor.service.ts L420-428
private async execCommand(command: string): Promise<{ ok: boolean; code: number; out: string }> {
  const exec = await window.zytrader.terminal.exec(command, '.');
  // 如果 terminal.exec 没有 timeout，可能永远 hang 住
}
```

**修复**: 使用 `Promise.race` 加超时（如 120s）。

---

#### O-16: 无单元测试覆盖
除 `command-system.spec.ts` 外，所有新服务没有测试。

---

#### O-17: `LoopCommandService.runCycle()` 是死代码
`LoopExecutorService.runOnce()` 是实际执行路径，`LoopCommandService.runCycle()` 从未被调用。

---

#### O-18: Debug Tab 视图缺少团队和发布信息
`buildLoopDebugViewModel()` 只展示基础 state 字段，不展示：
- 团队分配情况

## 四、操作手册

### 4.1 启动 Loop

```
/loop 实现用户登录页面
```

**完整语法**:
```
/loop <目标描述>
  [--team=<团队名>]           默认: auto（根据任务类型推断）
  [--task-type=<类型>]        development|testing|docs|ops|analysis|general
  [--max-iterations=<N>]      默认: 12
  [--every=<间隔>]            如 30s / 5m / 1h（暂未实现定时）
  [--auto-commit]             允许自动 git commit
  [--auto-push]               允许自动 git push
  [--require-review]          发布前必须人工审批
```

**示例**:
```
/loop 修复登录页样式问题
/loop 实现购物车功能 --team=dev --max-iterations=20
/loop 补充单元测试 --task-type=testing
/loop 分析系统瓶颈 --task-type=analysis
```

### 4.2 Loop 自动推断的任务类型

| 关键词 | 推断类型 | 默认团队 |
|--------|----------|----------|
| 页面/接口/实现/开发/功能/重构/fix/bug/feature/refactor/ui/api | `development` | `dev` |
| 测试/test/spec/case/coverage | `testing` | `test` |
| 文档/docs/readme/design | `docs` | `docs` |
| 部署/发布/运维/monitor/ops/ci/cd | `ops` | `ops` |
| 分析/调研/research/investigate | `analysis` | `general` |
| 其他 | `general` | `general` |

### 4.3 Loop 生命周期

```
┌─────────┐     ┌──────────┐     ┌───────────┐
│ planning │────▶│ executing │────▶│ verifying │
└─────────┘     └──────────┘     └───────────┘
                     │                 │
                     ▼                 ▼
               ┌───────────┐     ┌───────────┐
               │ repairing │◀────│  验证结果   │
               └───────────┘     └───────────┘
                     │                 │
              连续失败≥3              通过
                     │                 │
                     ▼                 ▼
               ┌─────────┐     ┌────────────────┐
               │ blocked  │     │ ready_for_review│
               └─────────┘     └────────────────┘
                                      │
                                      ▼
                              ┌──────────────────┐
                              │ ready_for_release │
                              └──────────────────┘
                                      │
                                      ▼
                              ┌───────────┐
                              │ completed  │
                              └───────────┘
```

### 4.4 开发任务强制流程

开发类任务（`taskType === 'development'`）必须遵循：

```
需求整理 → 需求文档 → 设计整理 → 设计文档 → /mode-dev 团队 → 代码实现 → 测试 → 验证 → 总结
```

如果缺少需求文档或设计文档，验证器会将 `missing-requirements-doc` / `missing-design-doc` 加入 blockers，阻止进入开发阶段。

### 4.5 验证矩阵

| 维度 | 检查内容 | 脚本 |
|------|----------|------|
| `compile` | Angular build | `npx ng build zyfront-desktop-web` |
| `ui` | HTML 模板存在性 + SCSS 存在性 + 空模板检测 | `npm run ui:check` |
| `api` | TypeScript 编译 + Angular 构建 | `npm run api:check` |
| `data` | src 目录可读性 + 配置文件合法性 | `npm run data:check` |
| `terminal` | UTF-8 验证 / npm test | `npm run verify:utf8` |

一键执行所有检查:
```bash
npm run check
```

### 4.6 修复策略

```
错误输出 → 错误分类
            ├── syntax  (优先级 100) → AST 结构化修补
            ├── type    (优先级 80)  → AST 结构化修补
            ├── lint    (优先级 60)  → 精确替换
            ├── unknown (优先级 50)  → 启发式修补
            ├── logic   (优先级 40)  → 启发式修补
            └── style   (优先级 20)  → 精确替换

修补路径：AST 结构化修补 → 启发式修补 → 失败升级

连续失败阈值：
  同一错误连续失败 3 次 → 升级为 blocked（暂停等待人工介入）
  最近 5 轮验证全部失败且推荐相同 → 标记"无进展"
  重试次数 ≥ 6 → 警告"可能需要人工介入"
```

### 4.7 发布门禁

发布前必须满足以下必要条件：

| 检查项 | 必要 | 说明 |
|--------|------|------|
| 功能实现完成 | ✅ | 存在 status=done 的 implementation 步骤 |
| 关键测试通过 | ✅ | 测试步骤完成 + 验证通过 |
| Build 成功 | ✅ | buildStatus=passed |
| 编译验证通过 | ✅ | verificationMatrix.compile.passed=true |
| 无已知阻塞项 | ✅ | blockedReasons 为空 |
| 变更摘要已记录 | ✅ | lastSummary 非空 |

触发审批的条件：
- 高风险文件变更（`*.config.ts` / `*.module.ts` / `main.ts` / `index.ts`）
- 变更文件数 > 10
